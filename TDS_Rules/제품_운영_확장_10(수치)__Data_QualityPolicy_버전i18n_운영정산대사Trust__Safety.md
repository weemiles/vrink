## 1) 데이터 품질(Data Quality)·데이터 계약(Data Contract) 운영

### 1.1 Source of Truth(SoT) 지정 규칙
- [운영 가드레일] 핵심 엔터티(유저/주문/결제/정산 등)는 SoT를 1개 시스템으로 고정한다.
- [운영 가드레일] SoT가 아닌 시스템의 값은:
    - 표시용 캐시로만 사용
    - 불일치 발견 시 SoT를 기준으로 자동 정정(가능하면)

### 1.2 데이터 계약(스키마) 변경 게이트
- [운영 가드레일] 아래 변경은 “breaking”으로 분류한다.
    - 필드 삭제
    - 타입 변경
    - enum/상태값 삭제
- [운영 가드레일] breaking 변경은:
    - schema_version +1
    - 구버전 호환 기간 30일
    - 변경 공지 채널 1개(문서/슬랙/노션)로 고정

### 1.3 데이터 검증 룰(수치)
- [운영 가드레일] 파이프라인/저장 시점에서 최소 아래를 검증
    - null 허용 여부
    - 범위(min/max)
    - 유니크(고유) 키 중복률
- [운영 가드레일] 품질 임계치
    - 핵심 테이블 중복률: 0.1% 미만
    - 결측률(null): 0.5% 미만 (필수 필드 기준)
    - 지연(데이터 갱신): 배치면 24시간 이내, 실시간이면 5분 이내

### 1.4 데이터 이상 탐지(알림) 기준
- [운영 가드레일] 전일 대비 변화율 경보:
    - 핵심 카운트가 -20% 이하
    - 실패율이 +1%p 이상
    - 중복률이 +0.2%p 이상

## 2) 콘텐츠/정책(Policy) 운영 시스템

### 2.1 정책 문서 버전 규칙
- [운영 가드레일] 정책/약관은 항상 vYYYY.MM.DD 형식으로 버전을 가진다.
- [운영 가드레일] 사용자 노출 시 “변경 요약”은:
    - 최대 5줄
    - 핵심 변경 항목 3개 이내

### 2.2 강제 재동의(리컨센트) 트리거(수치)
- [운영 가드레일] 아래가 해당되면 재동의 필요
    - 데이터 수집 목적 추가
    - 제3자 제공 범위 확대
    - 보관 기간 증가
- [운영 가드레일] 재동의 유예 기간:
    - 일반: 30일
    - 민감 변경: 7일

### 2.3 콘텐츠 변경 승인(4-eyes)
- [운영 가드레일] 법적/금전적 영향이 있는 문구는 2인 승인
    - 작성자 + 승인자(다른 사람)

## 3) 국제화(i18n) 운영 — 번역·품질·릴리즈

### 3.1 번역 키(Key) 규칙
- [운영 가드레일] 키 네이밍: domain.screen.element.state
    - 예: payment.review.cta.submit
- [운영 가드레일] 한 키에 문장 2개 이상 넣지 않는다.

### 3.2 언어별 길이 가드레일(수치)
- [운영 가드레일] 기본 언어 대비 확장 비율(문장 길이)
    - 영어/일본어: 1.3x
    - 독일어/러시아어: 1.6x
- 이 비율을 넘는 문자열은:
    - 문구 축약 버전(Short) 키를 추가

### 3.3 번역 QA 범위(릴리즈 게이트)
- [운영 가드레일] 매 릴리즈마다 최소 아래 화면은 언어 1개 이상으로 점검
    - 로그인/가입
    - 결제(또는 핵심 전환)
    - 설정
    - 오류/빈 상태

### 3.4 번역 릴리즈 정책(수치)
- [운영 가드레일] 새 언어 론칭은 점진 배포
    - 1% → 10% → 50% → 100%
- 단계 간 최소 관측:
    - 24시간

## 4) 결제/정산 ‘백오피스 운영’ — 정산 대사·실패/보류 처리

### 4.1 정산 주기/마감 규칙(예시 수치)
- [운영 가드레일] 정산 주기: 기본 주 1회 또는 월 1회로 고정
- [운영 가드레일] 마감 이후 수정은:
    - 관리자에서 직접 수정 금지
    - ‘정정 전표(Adjustment)’로만 처리

### 4.2 대사(Reconciliation) 필수 키
- [운영 가드레일] 대사 키는 최소 아래 3개를 포함
    - transaction_id
    - amount
    - status
- [운영 가드레일] 대사 불일치 허용률: 0.05% 미만

### 4.3 실패/보류 상태 전이 규칙
- [운영 가드레일] 상태는 “되돌아가지” 않게 단방향을 기본으로 설계
    - pending → succeeded / failed
    - failed → retry_pending(선택) → succeeded / failed
- [운영 가드레일] 재시도 횟수: 최대 2회

## 5) 고객 신뢰/안전(Trust & Safety) 운영

### 5.1 신고/차단 플로우 규칙
- [운영 가드레일] 신고 카테고리: 최대 6개
- [운영 가드레일] 신고 제출 필수값: 카테고리 + 사유(텍스트) 1개
- [운영 가드레일] 증빙 업로드:
    - 최대 3개
    - 장당 10MB 이하

### 5.2 제재 정책(수치)
- [운영 가드레일] 기본 제재 단계(에스컬레이션)
    - 1회: 경고
    - 2회: 기능 제한 24시간
    - 3회: 정지 7일
    - 4회: 영구 정지(필요 시)

### 5.3 오탐/이의제기(appeal) 운영
- [운영 가드레일] 이의제기 응답 목표: 72시간 이내
- [운영 가드레일] 이의제기 입력 필드: 최대 2개(사유 + 증빙)

---

## 6) 최소 실행 경로(적용 순서)
1) Data Quality: SoT 1개 지정 + breaking 변경 게이트(버전+30일)부터 고정한다.
2) Policy: 버전(vYYYY.MM.DD) + 재동의(7일/30일) 기준을 고정한다.
3) i18n: 키 규칙 + 길이 1.3x/1.6x 가드레일 + 핵심 화면 QA를 고정한다.
4) 정산: 마감 후 정정 전표 원칙 + 대사 키/불일치 0.05% 기준을 운영한다.
5) T&S: 신고 카테고리 6개 상한 + 제재 단계/시간(24h/7d)부터 적용한다.
