# 제품 운영 확장 2(수치) — A11y 실운용·정보구조(IA)·상태/데이터 모델·실험 해석·DS 기여


## 0. 범위와 중복 방지 선언
- 이 페이지는 컴포넌트 스펙/Props/기본값을 다루지 않습니다.
- 또한 이미 작성된 “운영 수치” 문서들과 주제/내용이 겹치지 않게 다음 범위만 다룹니다.
    - A11y는 “터치 타겟/대비/aria-label” 같은 기본 원칙(TDS 헌법 — Toss 모바일 앱 디자인 실무 가이드라인 (2026): 레이아웃·타이포·컬러·컴포넌트·접근성, TDS 컴포넌트 실무 노트 3(Mobile) — TextButton·IconButton·Result·Slider)을 재작성하지 않고, 스크린리더/포커스/동적 읽힘처럼 ‘실운용 규칙’만 정리합니다.
    - 내비게이션/비동기/표기/보안/Analytics 수치 자체는 제품 운영 심화(수치) — 내비게이션·비동기/캐시·표기(Formatting)·보안 UX·Analytics에 이미 있으므로, 여기서는 측정 설계/실험 해석/데이터 모델/정보구조 같은 상위 운영 규칙으로만 확장합니다.
    - QA/변경관리/카피/i18n/성능 UX 규칙은 제품 품질/운영 규칙(수치) — QA·변경관리·카피·i18n·성능 UX에 이미 있으므로, 여기서는 그 다음 레벨(정의·운영·해석·협업 템플릿)만 다룹니다.

## 1) 접근성(A11y) 실운용 — 스크린리더/포커스/동적 콘텐츠 규칙

### 1.1 스크린리더 라벨 품질 기준(수치)
- 한 요소의 읽기(라벨) 길이:
    - 권장 10~30자
    - 최대 45자 (45자 초과면 정보 구조를 분리)
- 라벨은 “명사+행동” 순서를 고정합니다.
    - 예: 계좌번호 복사, 약관 자세히 보기

### 1.2 포커스 순서(탐색) 규칙
- 포커스 이동은 기본적으로 시각적 읽기 순서(좌→우, 상→하)와 동일해야 합니다.
- 예외 허용 기준:
    - 화면 상단 고정 CTA가 있을 때만(최대 1개) 포커스 순서를 앞당길 수 있음.

### 1.3 모달/바텀시트 포커스 트랩 규칙(수치)
- 오버레이가 열리면:
    - 최초 포커스는 오버레이 내부의 “가장 중요한 제목” 또는 “첫 입력”으로 이동(둘 중 1)
    - 오버레이 외부로 포커스가 나가면 0회여야 합격
- 닫힐 때:
    - 포커스는 오버레이를 연 트리거로 복귀
    - 복귀 실패 시: 화면 최상단(타이틀)로 이동

### 1.4 동적 콘텐츠 읽힘 규칙(로딩/성공/실패)
- 로딩:
    - 로딩이 2초를 넘는 경우, 스크린리더용 상태 텍스트를 1회만 제공(반복 읽힘 금지)
- 성공/실패:
    - 토스트/배너로 상태를 알리면, 스크린리더 공지 역시 최대 1회
    - 10초 내 동일 이벤트(동일 메시지)는 재공지 금지

### 1.5 입력 오류(Validation) 접근성 규칙
- 오류는 “필드 단위”로 연결되어야 합니다.
- 규칙(수치):
    - 한 필드에 오류 메시지는 1개만
    - 오류 메시지는 최대 40자
    - 오류가 있는 필드가 2개 이상이면, 화면 상단에 오류 요약을 1개 제공

### 1.6 접근성 회귀 체크(최소 세트)
- 기능 변경 시, 최소 5항목을 반드시 확인합니다.
    1) 스크린리더 읽기 순서
    2) 오버레이 포커스 트랩
    3) 동적 상태(로딩/성공/실패) 공지 중복
    4) 입력 오류 연결
    5) 탐색 시 “중복 라벨(같은 문장 반복)” 여부

## 2) 정보구조(IA) / 콘텐츠 구조 템플릿 — 복잡 화면을 구조로 고정

### 2.1 화면을 섹션으로 쪼개는 규칙(수치)
- 한 화면의 주요 섹션 개수:
    - 권장 3~5개
    - 6개 이상이면 화면을 2단계로 분리(탭/아코디언/서브화면)
- 한 섹션 안의 정보 행(row) 개수:
    - 권장 5~12개
    - 12개 초과 시: 검색/필터/요약 영역 추가(택1)

### 2.2 ‘요약 → 상세’ 계층 고정 규칙
- 상단 요약 카드(가능하면):
    - 핵심 숫자/상태는 최대 3개
    - 다음 행동(CTA)은 최대 1개
- 상세 정보는 섹션별로 나누고, “반복되는 라벨”은 표(TableRow류)로 통일(스펙 자체는 기존 문서 참조)

### 2.3 설정 화면 템플릿(규칙)
- 설정은 4그룹을 넘기지 않습니다.
    1) 계정
    2) 보안
    3) 알림
    4) 기타
- 각 그룹당 항목은 최대 8개
- 8개 초과 시: ‘상세 설정’으로 2단계 분리

### 2.4 결제/신청 플로우 템플릿(수치)
- 단계 수:
    - 권장 3~5단계
    - 6단계 이상이면 중간 저장/재개 기능 또는 단계 통합 검토
- 각 단계는 “입력 요소”를 제한합니다.
    - 한 단계당 입력(텍스트필드/선택) 최대 7개

### 2.5 약관/고지 템플릿(규칙)
- 약관 체크는 ‘전체 동의’가 있으면 다음을 고정:
    - 필수 항목: 최대 3개 (초과하면 묶어서 ‘필수 항목 더보기’)
    - 선택 항목: 최대 5개

## 3) 상태/데이터 모델(프론트-백 연결) — UI가 망가지지 않는 설계 규칙

### 3.1 화면 상태 머신 최소 모델
- 모든 화면은 최소 4상태를 갖습니다(명칭 고정).
    1) idle (초기/대기)
    2) loading
    3) success
    4) error
- 리스트/피드 화면은 2상태를 추가합니다.
    - refreshing (당겨서 새로고침)
    - paginating (무한 스크롤)

### 3.2 로딩을 ‘중첩’하지 않는 규칙(수치)
- 한 화면에서 동시에 보이는 로딩 표현은 최대 1종
    - 예: 스켈레톤 + 스피너 동시 노출 금지
- 로딩이 중첩되어야만 하는 특수 케이스는 예외로 두되:
    - 상단 글로벌 로딩 1개 + 리스트 아이템 인라인 로딩은 최대 3개 아이템까지만

### 3.3 API 응답 계약(Contract) 규칙
- 프론트에서 ‘조건 분기’가 폭발하는 패턴을 금지합니다.
- 규칙:
    - nullable 필드(없을 수 있음)는 응답에 명시적으로 null로 내려온다(키 생략 금지)
    - 배열은 비어있으면 []
    - 날짜/시간은 ISO 8601만

### 3.4 에러 코드 체계(최소 세트)
- 최소 6종으로 통일(예외는 추가하되 분류는 이 안에서)
    - NETWORK
    - TIMEOUT
    - UNAUTHORIZED
    - FORBIDDEN
    - VALIDATION
    - UNKNOWN
- 화면은 에러 코드별로 “다음 행동”을 1개만 제공
    - 예: NETWORK/TIMEOUT → 다시 시도
    - UNAUTHORIZED → 로그인
    - FORBIDDEN → 권한 안내

### 3.5 데이터 포맷팅 책임 분리
- UI 컴포넌트 안에서 포맷팅 금지
- 포맷터 레이어에서 다음만 책임짐:
    - 금액/퍼센트/날짜
    - 마스킹
    - 로케일
> 포맷 규칙의 세부 수치는 제품 운영 심화(수치) — 내비게이션·비동기/캐시·표기(Formatting)·보안 UX·Analytics의 표기 섹션을 따른다.

## 4) 실험/측정(Analytics) 심화 — 이벤트가 아니라 ‘지표/해석’ 규칙

### 4.1 메트릭 설계 최소 세트
- 실험 1개는 최소 3종 메트릭을 고정합니다.
    1) Primary: 핵심 성과 1개
    2) Guardrail: 악화되면 중단하는 안전 지표 1개
    3) Diagnostic: 원인 추적 1개

### 4.2 최소 관측 기간/표본(수치)
- 관측 기간:
    - 최소 7일, 권장 14일
- 표본 기준(대략 규칙):
    - 전환율 변화 5%p 미만을 보고 싶으면, 각 그룹에 최소 1,000 유효 세션 확보
    - 1,000이 안 되면 기간을 늘리고, 결론은 “보류”를 기본

### 4.3 중단(Stop) 규칙
- Guardrail이 악화되면 즉시 중단
    - 크래시율 +0.2%p 이상
    - 결제/인증 실패율 +1.0%p 이상
- Primary가 좋아도 Guardrail 악화면 “실험 실패”로 기록

### 4.4 세그먼트(필수 분해) 규칙
- 결과는 최소 3세그먼트로 반드시 분해합니다.
    - 신규 vs 기존
    - iOS vs Android
    - 저속 네트워크(느림 케이스) vs 정상

### 4.5 실험 중 변경 금지 규칙
- 실험 기간 중 UI/문구/플로우 추가 변경 금지
- 예외는 크리티컬 버그 뿐이며, 예외 적용 시 해당 실험은 “무효”로 마킹

## 5) 디자인 시스템 운영/기여(Contribution) — 문서/리뷰를 템플릿화

### 5.1 기여 제안서(Proposal) 최소 템플릿
- 새 컴포넌트/변경 제안은 1페이지로 고정하고, 항목은 8개를 넘기지 않습니다.
    1) 문제(1문장)
    2) 사용자 영향(누가/어디서)
    3) 기존 대안(왜 부족한지)
    4) 제안 솔루션(요약)
    5) 상태(기본/pressed/disabled/loading 등) 목록
    6) 접근성(읽힘/포커스)
    7) 호환성/마이그레이션
    8) 측정/QA 포인트

### 5.2 리뷰 게이트(수치)
- PR/변경 1개는 최소 2승인을 원칙으로 합니다.
    - 디자인 1
    - 개발 1
- 고위험 변경(동작/접근성/결제/인증)인 경우:
    - 추가 승인 1(총 3)

### 5.3 디프리케이션(폐기) 규칙
- Deprecated 공지 기간: 4주
- 4주 후 제거 가능하되, 제거 전 체크:
    - 사용 화면 0개 확인
    - 마이그레이션 가이드 링크 1개 제공

### 5.4 릴리즈 노트 최소 규칙
- 릴리즈 노트는 항목당 3줄 제한
    - 무엇이 바뀌었는지
    - 누가 영향 받는지
    - 무엇을 해야 하는지

## 6) 적용 순서(최소 실행 경로)
1. 화면별 상태 머신(idle/loading/success/error)을 이름 그대로 통일한다.
1. 에러 코드 6종으로 매핑하고, 화면의 다음 행동을 1개로 고정한다.
1. 설정/신청 플로우 템플릿의 “개수 제한”을 먼저 적용한다.
1. 스크린리더 회귀 체크 5항목을 릴리즈 체크리스트에 추가한다.
1. 실험은 Primary/Guardrail/Diagnostic 3종을 강제하고 7일 미만 결론을 금지한다.
1. DS 기여는 Proposal 8항목 템플릿으로 통일한다.
