## 1) Input(입력) 실무 운영 규칙 — “검증/에러/키보드/포커스”를 수치로 고정
> 컴포넌트 선택(TextField/TextArea/SplitTextField/SearchField/Keypad)은 기존 문서를 따르고, 이 섹션은 운영 정책을 고정합니다.

### 1.1 입력 길이(최소 기준)
- 이름/일반 텍스트: max 30자
- 주소/상세 설명: max 200자
- 검색어: max 50자 (이미 예시로도 자주 쓰는 값)
- 인증코드/핀: 고정 길이 6자리
- 주민번호/정부ID 계열: 고정 길이(예: 13자리) + 마스킹 기본 ON

### 1.2 검증 타이밍 규칙(3단계)
- 입력 중(onChange):
    - 형식 보정(formatting)만 허용
    - 에러 메시지 노출 금지(사용자 스트레스 방지)
- 포커스 아웃(onBlur):
    - 형식 검증(정규식/자리수) + 에러 노출
    - 에러는 1개만 노출(가장 우선순위 높은 것)
- 제출(submit):
    - 서버 검증 포함 최종 검증
    - 제출 실패 시 첫 에러 필드로 스크롤/포커스 이동

### 1.3 에러 메시지(수치/규칙)
- 한 필드에서 동시에 보여주는 문장 수: 최대 1문장
- 길이: 최대 30자 권장
- 에러가 2개 이상이면:
    - (1) “필수”를 최우선
    - (2) “형식”
    - (3) “범위/정책” 순

### 1.4 자동 포맷(Formatting) 규칙
- 입력값이 숫자/금액이면:
    - 표시: toLocaleString()
    - 저장/전송: 콤마 제거한 원시 숫자
- 전화번호:
    - 표시: 3-4-4
    - 저장: 숫자만
- 카드/계좌/민감번호:
    - 표시: 일부 마스킹
    - 저장/로그: 원문 금지 (보안 규칙은 제품 운영 확장 3(수치) — 보안/리스크·컴플라이언스·장애대응·CS·성장 참조)

### 1.5 키보드/포커스(모바일 필수 규칙)
- 다음 필드로 이동:
    - 키보드 Next/Enter → 다음 필드로 포커스 이동을 기본
- 단일 페이지 내 입력 필드 수가 7개 이상이면:
    - 그룹(섹션)으로 분리
    - ‘다음’ CTA를 단계형으로
- 입력 화면에서 하단 CTA가 필요한 경우:
    - BottomCTA 규칙(2절)을 따른다

### 1.6 삭제/초기화 UX 규칙
- “내용 초기화”는 다음 중 1개로 표준화:
    - Clear 버튼(필드 내부)
    - ‘전체 지우기’ CTA(필드가 3개 이상인 폼)
- 전체 지우기는 확인 다이얼로그 없이 즉시 수행 (되돌리기 UX를 제공할 수 있으면 더 좋음)

## 2) BottomCTA / FixedBottomCTA — Safe Area·스크롤·키보드 충돌을 수치로 막기
> 컴포넌트 사용법/Props는 기존 문서(예: BottomCTA 이해하기)를 보고, 여기서는 적용 규칙만 고정합니다.

### 2.1 Bottom CTA 사용 조건
- 다음 중 1개 이상이면 Bottom CTA를 사용합니다.
    - 화면 길이가 길어 스크롤이 2뷰포트 이상
    - 완료/다음이 핵심 전환(결제, 신청, 저장)
    - 입력 중에도 CTA 접근이 필요

### 2.2 버튼 개수 규칙
- CTA 1개가 원칙(Primary 1개)
- CTA 2개는 예외로만 허용:
    - “취소/다음” 같이 대등한 선택이 필요할 때
    - 혹은 법적 동의 등으로 ‘거절’이 반드시 있어야 할 때

### 2.3 스크롤 hide/show 규칙
- hideOnScroll은 다음 조건에서만 켭니다.
    - 콘텐츠가 길고, CTA가 콘텐츠를 가린다
    - 스크롤이 아래로 내려가는 동안 CTA가 방해가 된다
- hide/show의 최소 임계값:
    - “작은 떨림”으로 숨기지 않기 위해 24px 이동 후 hide/show를 트리거(권장)
    - (문서 기본값 1px을 그대로 쓰지 말고, 실무에선 24px로 올리는 것을 기본 규칙으로 둔다)

### 2.4 키보드 충돌 방지 규칙
- 입력 화면(폼)에서는:
    - CTA가 반드시 남아야 하면: 키보드 위 고정
    - CTA가 방해가 되면: CTA 숨김 + 상단/중단에 ‘다음’ 버튼 제공
- “Double CTA”는 키보드 위 고정이 제한될 수 있으므로:
    - Double이 필요하면, 입력 단계에서는 Single로 설계하고 마지막 단계에서 Double을 사용

### 2.5 Safe Area 하단 여백(수치)
- 하단 여백은 최소 20px
- 홈 인디케이터가 있는 디바이스(iOS)는 Safe Area를 포함해 34px 이상을 보장

## 3) 토글/선택(Select) — 상태 모델 + 접근성 + 실수 방지를 수치화
> Switch/SegmentedControl/Radio/Checkbox의 Props는 기존 문서가 있고, 여기서는 운영 규칙을 정의합니다.

### 3.1 기본 원칙: 선택은 “기본값”을 반드시 가진다
- SegmentedControl/Radio:
    - 기본값이 없으면 UI가 불안정해지고 분석도 흔들리므로
    - default 선택 1개를 반드시 지정

### 3.2 옵션 개수 제한(수치)
- SegmentedControl:
    - 옵션 2~5개 권장
    - 6개 이상이면 dropdown/필터 화면으로 전환
- Radio(단일 선택):
    - 옵션 2~7개
    - 8개 이상이면 검색/필터/그룹핑 필요
- Checkbox(복수 선택):
    - 한 화면에서 ‘복수 선택’ 항목이 10개 이상이면
        - 카테고리/접기(토글)/검색 중 최소 1개 제공

### 3.3 기본 CTA 문구 규칙
- 토글/선택의 라벨은 “켜짐/꺼짐” 같은 상태 단어를 포함하지 않는다.
- 라벨은 대상(무엇을)만 말하고, 상태는 컨트롤이 말하게 한다.

### 3.4 파괴적/비가역 선택(규칙)
- 삭제/탈퇴/영구 변경 등은:
    - Checkbox로 바로 실행하지 않는다
    - ConfirmDialog 또는 2단계 확인으로 올린다
- Radio 선택으로 즉시 서버 반영되는 설정은:
    - 선택 후 서버 실패 시 롤백이 보이게 하거나
    - 저장 CTA로 분리한다

## 4) 디자인 시스템(DS) 운영/문서화 — “바뀔 때 망하지 않게” 하는 규칙

### 4.1 변경 요청(컴포넌트/토큰) 최소 패키지
- 변경 1건당 아래 6가지를 문서/티켓에 고정합니다.
    1) 변경 이유(문제)
    2) 사용자 영향(스크린샷/시나리오)
    3) 호환성(깨지는 화면)
    4) 마이그레이션 방법
    5) 릴리즈 노트 문장(1~2줄)
    6) 롤백 플랜

### 4.2 Deprecation(제거 예정) 운영 수치
- deprecate 발표 → 제거까지 최소 2 릴리즈 유예
- 앱이 스토어 심사/업데이트 주기가 길면 최소 4주 유예

### 4.3 가이드 품질 체크(체크리스트)
- 새 가이드는 최소 아래 8개를 포함:
    - 언제 쓰는가
    - 언제 쓰지 않는가
    - 최소/최대 옵션(최악 케이스)
    - 상태(기본/disabled/loading/error)
    - 접근성
    - 분석 이벤트(필요 시)
    - 성능/지연 주의사항(필요 시)
    - QA 체크

## 5) Chart(예: RN BarChart) — 데이터 구조/해석 규칙을 먼저 고정
> 컴포넌트 Props를 외우기 전에 “데이터가 어떤 의미인지”가 먼저 고정되어야 운영이 됩니다.

### 5.1 데이터 스케일 규칙
- 차트 단위는 1개만(예: 원, %, 횟수)
- 단위를 섞으면 오해가 생기므로 금지

### 5.2 축/라벨 규칙(수치)
- x축 라벨:
    - 한 화면에서 최대 7개만 표시
    - 8개 이상이면 스크롤/집계(주간→월간) 중 1개 적용
- y축 값:
    - 반올림 규칙을 고정(예: 천 단위, 만 단위)
    - 소수점은 최대 1자리

### 5.3 빈 데이터/결측 규칙
- 데이터가 0이거나 없음:
    - “0”과 “없음”을 구분(예: 회색 막대/점선)
    - 사용자에게 이유를 1문장으로 제공

### 5.4 해석 가드레일
- 변동이 작은 지표는
    - y축을 0에서 시작하지 않으면 과장될 수 있으므로
    - 기본은 0 기준
- 0 기준을 포기해야 하면:
    - 범위 축소 이유를 화면에 명시

## 6) 적용 순서(최소 실행 경로)
1. 폼 1개를 골라 입력 길이(max), 검증 타이밍(onChange/onBlur/submit)을 고정한다.
1. 입력 화면이면 BottomCTA의 스크롤/키보드 규칙(24px 임계값, 단계 분리)을 적용한다.
1. 선택 UI는 옵션 개수 제한(2~5, 2~7, 10+) 규칙으로 컴포넌트를 결정한다.
1. DS 변경 요청은 6항목 패키지로만 받도록 운영한다.
1. 차트는 ‘단위 1개, 라벨 7개, 소수점 1자리’ 규칙부터 먼저 적용한다.
