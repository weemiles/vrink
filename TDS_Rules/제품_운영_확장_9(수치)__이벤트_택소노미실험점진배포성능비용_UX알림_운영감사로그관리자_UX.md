- 이 페이지는 컴포넌트 Props/기본값/허용값을 재작성하지 않습니다.

    - TDS 전체 헌법: TDS 헌법 — Toss 모바일 앱 디자인 실무 가이드라인 (2026): 레이아웃·타이포·컬러·컴포넌트·접근성
    - 오버레이/토스트/에셋/로딩: TDS 심화 학습 — 오버레이·토스트·에셋·스켈레톤·TableRow (원문 구조 기반)
    - 실무 컴포넌트 확장: TDS 컴포넌트 실무 노트 2 — SearchField·Loader·Menu/Dropdown·ListRow·Keypad·Carousel·NumericSpinner·BottomInfo
    - 모션·딥링크/푸시·오프라인·권한 UX·A11y·플래그 운영: 제품 운영 확장 5(수치) — 모션·딥링크/푸시·오프라인·권한 UX·A11y 시나리오·Server-driven UI
    - 테마·폼·차트·금융 리스크·Observability: 제품 운영 확장 7(수치) — Theming/다크모드·Form 검증·차트 해석·금융 리스크 UX·Observability

    - [운영 가드레일]: 팀이 합의해 고정해야 하는 수치/정책

## 1) 데이터/이벤트 택소노미 운영 (Analytics 설계 규칙)

### 1.1 이벤트 네이밍 규칙
- [운영 가드레일] 이벤트 이름은 snake_case로 통일한다.
- [운영 가드레일] 접두사로 도메인을 고정한다.
    - 예: auth_login_success, payment_submit_click, search_query_submit
- [운영 가드레일] “동사”는 1개만 쓴다.
    - 허용: view, click, submit, success, fail, dismiss

### 1.2 이벤트 속성(property) 규칙(스키마)
- [운영 가드레일] 이벤트 1개당 속성 개수: 최대 12개
- [운영 가드레일] 필수 공통 속성(최소 6개)
    - screen_name
    - flow_id (동일 플로우를 묶는 ID)
    - step_index (0부터)
    - variant (실험/플래그)
    - is_logged_in (true/false)
    - app_version

### 1.3 PII(개인정보) 금지 규칙
- [운영 가드레일] 이벤트/로그 속성에 아래 값은 직접 저장 금지
    - 이메일 원문
    - 전화번호 원문
    - 계좌/카드 전체
    - 이름/주소
- [운영 가드레일] 꼭 필요하면:
    - 해시(일방향) + salt(서버) 방식으로 변환한 식별자만 사용

### 1.4 이벤트 버전 관리(수치)
- [운영 가드레일] 모든 이벤트는 schema_version을 포함한다.
- [운영 가드레일] breaking change는 schema_version을 +1.
- [운영 가드레일] 구버전 스키마 수집 유지 기간: 30일

### 1.5 계측 품질 기준(수치)
- [운영 가드레일] 핵심 이벤트 누락률: 1% 미만
- [운영 가드레일] 중복 발생률(같은 flow_id/step에서 동일 이벤트): 0.5% 미만

## 2) 실험(AB)·점진 배포(Gradual rollout) 운영 규칙

### 2.1 샘플링/기간 최소치(권장 수치)
- [운영 가드레일] AB 테스트 최소 표본:
    - 각 그룹 최소 1,000명 또는
    - 최소 기간 7일 중 더 큰 조건을 충족

### 2.2 실험 격리(오염 방지) 규칙
- [운영 가드레일] 할당 단위는 기본적으로 user_id (로그인 전이면 device_id)
- [운영 가드레일] 같은 유저가 다중 기기에서 그룹이 달라지면 결함으로 처리

### 2.3 가드레일 지표(실험 중단 기준)
- [운영 가드레일] 아래 중 하나라도 충족 시 실험 즉시 중단 또는 롤백
    - crash-free users -0.2%p 하락
    - p95 latency +30% 상승
    - 결제 실패율 +0.5%p 상승 (해당 시)
    - API error rate +1%p 상승

### 2.4 점진 배포 단계(수치)
- [운영 가드레일] 기본 램프업: 1% → 5% → 25% → 50% → 100%
- [운영 가드레일] 단계 간 관측 시간:
    - 최소 30분 (에러/크래시)
    - 가능하면 24시간 (전환/이탈)

## 3) 성능/네트워크 비용 UX 운영 (성능을 정책으로)

### 3.1 로딩 체감 기준(수치)
- [운영 가드레일] 사용자 액션 후 피드백:
    - 100ms 이내: 즉시 반응
    - 100~1000ms: inline 피드백(pressed/스피너)
    - 1초 초과: 로딩 상태(문구 포함)
    - 2초 초과: 진행 문구 + 취소(가능하면)

### 3.2 이미지 정책(권장 수치)
- [운영 가드레일] 썸네일:
    - 긴 변 최대 320px
    - 파일 크기 150KB 이하
- [운영 가드레일] 본문 이미지:
    - 긴 변 최대 1280px
    - 파일 크기 500KB 이하

### 3.3 동영상 정책(권장 수치)
- [운영 가드레일] 자동재생은 기본적으로 Wi‑Fi에서만
- [운영 가드레일] 셀룰러에서 동영상은:
    - 사용자가 재생을 명시적으로 눌러야 한다.
    - 데이터 사용 안내(1줄)를 제공한다.

### 3.4 리스트/피드 성능 가드레일
- [운영 가드레일] 무한 스크롤은:
    - 초기 로드 20개
    - 추가 로드 10~20개 단위
    - “더 보기” 클릭형 전환 조건: 실패가 잦거나 로딩이 2초를 자주 초과할 때

## 4) 알림/메시징(푸시·인앱) 운영 규칙

### 4.1 빈도 캡(Frequency cap)
- [운영 가드레일] 푸시:
    - 하루 최대 2회
    - 주 최대 8회
- [운영 가드레일] 인앱(배너/토스트 등):
    - 같은 메시지 반복 노출은 하루 최대 1회

### 4.2 조용한 시간(Do not disturb)
- [운영 가드레일] 기본 조용한 시간: 22:00~08:00
- 예외(긴급/보안 알림)는 허용하되, 사용자 설정으로 끌 수 있어야 함.

### 4.3 중복 제거 규칙
- [운영 가드레일] 같은 이벤트로 알림이 2개 이상 발생하면:
    - 가장 중요한 1개만 전송
    - 나머지는 묶어서 인앱 센터에 기록

### 4.4 측정(알림 성과 지표)
- [운영 가드레일] 필수 계측 이벤트
    - push_delivered
    - push_opened
    - push_dismissed (가능하면)
    - 딥링크 도착 deeplink_landed

## 5) 로그/감사(Audit)·관리자(Admin) UX 운영 규칙

### 5.1 감사로그 필수 필드
- [운영 가드레일] 모든 변경 이벤트는 아래를 저장
    - who: actor_id
    - when: timestamp
    - what: resource_type + resource_id
    - before/after: 변경된 필드만(diff)
    - where: client/app_version

### 5.2 롤백/승인 정책(수치)
- [운영 가드레일] 고위험 변경(가격/권한/정산)은:
    - 2인 승인(4-eyes) 적용
- [운영 가드레일] 롤백 가능 기간:
    - 최소 7일

### 5.3 관리자 작업의 실수 방지 규칙
- [운영 가드레일] 파괴적 액션(삭제/정산 확정)은:
    - 확인 단계 2번(ConfirmDialog + 최종 입력) 또는
    - 입력 확인(예: “DELETE” 입력) 중 하나

## 6) 최소 실행 경로(적용 순서)
1) 이벤트: 네이밍 + 공통 6필드 + PII 금지부터 고정한다.
2) 실험/배포: 1-5-25-50-100 램프업 + 중단 기준(크래시/지연/에러/실패율)을 고정한다.
3) 성능: 1초/2초 로딩 체감 기준 + 이미지/비디오 정책을 고정한다.
4) 알림: 빈도 캡(2/8) + 조용한 시간(22-08)을 운영한다.
5) 감사로그: who/when/what/before-after 필드와 2인 승인(고위험)을 적용한다.
