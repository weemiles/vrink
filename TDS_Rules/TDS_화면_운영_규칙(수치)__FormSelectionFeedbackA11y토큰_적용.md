# TDS 화면 운영 규칙(수치) — Form·Selection·Feedback·A11y·토큰 적용


## 0. 이 페이지의 목적 (중복 방지)
- 이 페이지는 기존에 정리된 “컴포넌트별 Props/기본값”을 다시 쓰지 않습니다.
- 대신, 여러 컴포넌트를 묶어서 실제 화면에 적용할 때 필요한 운영 규칙 + 수치(간격/타이밍/길이/오프셋)만 정리합니다.
- 아래 문서의 내용(스펙/Props/토큰 표)은 여기서 복제하지 않고 필요 시 링크로만 참조합니다.
    - 메인 헌법(기본 간격/터치/타이포/컬러/버튼 등): TDS 헌법 — Toss 모바일 앱 디자인 실무 가이드라인 (2026): 레이아웃·타이포·컬러·컴포넌트·접근성
    - 오버레이/토스트/에셋/로딩/Skeleton/TableRow: TDS 심화 학습 — 오버레이·토스트·에셋·스켈레톤·TableRow (원문 구조 기반)
    - SearchField/Loader/Menu/ListRow/Keypad/Carousel 등: TDS 컴포넌트 실무 노트 2 — SearchField·Loader·Menu/Dropdown·ListRow·Keypad·Carousel·NumericSpinner·BottomInfo
    - TextButton/IconButton/Result/Slider: TDS 컴포넌트 실무 노트 3(Mobile) — TextButton·IconButton·Result·Slider

## 1. Form(입력/검증) — 실무 기본 수치 세트
> 목표: “폼 화면을 만들 때 매번 새로 정하지 않도록” 최소 기준 수치를 고정

### 1.1 레이아웃/간격(폼 전용)
- 필드(입력 컴포넌트) 사이 기본 간격: 12px
- 필드 그룹(섹션) 사이 간격: 24px
- 라벨 ↔ 입력칸 간격: 8px
- 입력칸 ↔ 헬퍼/에러 텍스트 간격: 6px
- 페이지 좌우 마진은 헌법의 16px를 그대로 사용(여기서 재정의하지 않음).

### 1.2 텍스트 길이 제한(문구 설계)
- 라벨(필드명): 최대 12자 권장
- 헬퍼 텍스트: 최대 28자 권장
- 에러 메시지: 최대 32자 권장
    - 더 길어지면 “문구”가 아니라 “설명”이므로 별도 안내(예: Dialog/BottomSheet/Result)로 올립니다.

### 1.3 입력 검증 타이밍(UX 고정 규칙)
- onChange 즉시 검증(typing 중): 금지(기본값)
    - 예외: 숫자 포맷팅(천 단위 콤마)처럼 즉시 피드백이 오히려 안정감을 주는 케이스
- onBlur 검증: 기본
- 제출(CTA) 클릭 시 전체 검증: 필수

### 1.4 Debounce / Throttle (검색/자동완성)
- 검색/자동완성 요청 debounce: 250ms
- 서버 호출이 비싼 경우(추천/주소검색 등): 350ms
- 최소 글자 수(min chars) 가드:
    - 일반 검색: 2자 이상부터 요청
    - 계정/고유값(이메일/아이디): 1자 이상도 가능

### 1.5 중복 제출 방지(결제/저장)
- 제출 버튼을 누른 직후 즉시 disabled 처리
- 서버 응답이 느린 케이스는 로딩 UI를 0.2초(200ms) 이후부터 표시
    - 이유: 0~200ms 구간에 로딩이 깜빡이면 오히려 느려 보임
- 타임아웃(UX 기준) 제안:
    - 3초: “조금 느림”
    - 7초: “명확한 안내 필요”(토스트/상단 메시지)
    - 12초: “실패 처리 + 재시도 제공” 권장

## 2. Selection Controls(선택) — 상태/규칙 수치

### 2.1 단일 선택 vs 다중 선택
- 단일 선택(라디오/세그먼트): 기본값을 정할 수 없으면 “선택하지 않음” 상태를 먼저 제공
- 다중 선택(체크): 기본값은 “모두 해제”를 원칙으로, 사용자가 켜도록 둠

### 2.2 토글(Switch) 즉시 반영 규칙
- 토글을 켜는 순간 서버 반영이 필요한 경우:
    - optimistic update 허용 기준: 실패 확률이 낮고, 실패 시 되돌릴 수 있을 때
    - 실패 시: 토스트 1회 + 원상복구
- 서버 요청이 필요한 토글의 로딩 표시는 스위치 자체에 붙이기 어렵다면, 해당 row 오른쪽에 Loader small을 1초 이내로만 노출(길어지면 별도 상태 UI)

### 2.3 선택 리스트(설정 화면) 최소 터치 영역
- 최소 터치 타겟은 헌법의 44x44px 규칙을 그대로 따름(여기서 재정의하지 않음).
- row 간 구분이 약하면 row 사이 여백을 늘리는 대신, 구분선(indented/full) + padding 유지로 해결

## 3. Feedback(피드백) 선택 트리 + 수치
> 이미 Toast/Overlay/Result의 컴포넌트 스펙은 다른 페이지에 있으므로, 여기서는 “언제 무엇을 쓰는가”를 수치로 고정

### 3.1 로딩
- 0~200ms: 로딩 UI를 보이지 않게(가능하면)
- 200ms~2s: Loader/Skeleton 중 선택
    - 레이아웃이 고정돼야 하면 Skeleton
    - 단순 대기/처리면 Loader
- 2s 이상: 진행 상황/다음 행동(취소/뒤로) 힌트를 제공

### 3.2 토스트
- 토스트는 “확인 버튼 없는” 단순 알림이 기본
- 버튼이 있는 토스트는 본문 페이지의 기본값(버튼 있으면 duration이 길어짐)을 감안해, 버튼이 정말 필요한 경우에만 사용
- 하단 고정 CTA가 있는 화면에서 bottomOffset은 기본값에서 +34px(홈 인디케이터 높이)만큼 추가를 1차 검토
    - 실제 값은 화면 구조에 따라 조정

### 3.3 다이얼로그/바텀시트
- 사용자가 결정을 내려야 하는 경우만 사용
- dimmer click으로 닫히면 안 되는 흐름(결제/삭제/중단 등)은 기본값에 기대지 말고 명시적으로 잠그기

## 4. A11y(접근성) — 필수 필드/라벨 수치

### 4.1 aria-label 작성 규칙(길이)
- aria-label은 10~30자 범위를 목표
    - 너무 짧으면 맥락 부족
    - 너무 길면 핵심 파악이 어려움
- 형식 템플릿(권장): “대상 + 행동 + 결과”
    - 예: “다음 항목 더 불러오기”, “검색어 삭제하기”, “설정 화면 열기”

### 4.2 에러/헬퍼 텍스트 대비
- 대비 기준은 헌법의 WCAG 원칙을 따르되, 실무 체크용으로:
    - 에러 텍스트는 “회색 계열”로 낮추지 말고, 상태 토큰(semantic) 기반 컬러를 사용
    - 에러 텍스트가 2줄을 넘어가면 UI가 깨지기 쉬우므로, 2줄 이상이면 안내 방식을 상향(Result/Dialog) 검토

## 5. 토큰/테마 적용 — 하드코딩 금지 영역(실무 규칙)

### 5.1 금지 리스트
- 금지: 임의 HEX로 버튼/링크/상태(성공/경고/에러) 색상 정의
- 금지: 폰트 사이즈(px) 직접 하드코딩
- 금지: opacity 임의 계산(가능하면 greyOpacity 토큰 사용)

### 5.2 허용 리스트(예외)
- 예외적으로 허용: 프로토타입 단계에서만, 배경 이미지 위 텍스트 가독성 확보를 위한 임시 overlay
    - overlay opacity 기본 시작값: 0.18
    - 필요 시: 0.31까지(그 이상이면 콘텐츠가 가려짐)

## 6. 실무 적용 “빠른 프리셋”

### 6.1 검색 화면
- SearchField fixed: 필요하면 사용
- 입력 debounce: 250ms
- 결과 로딩:
    - 200ms 이하면 로딩 미표시
    - 200ms 이상이면 Skeleton 또는 Loader

### 6.2 설정 화면(리스트)
- row 구분선: indented 기본
- 토글은 즉시 반영(가능할 때)
- 실패 시 토스트 1회 + 원복

### 6.3 결제/삭제 같은 위험 액션
- 토스트로 끝내지 말고 Confirm 계열 사용
- 중복 제출 방지: 즉시 disabled
- 12초 초과면 실패 처리 + 재시도
