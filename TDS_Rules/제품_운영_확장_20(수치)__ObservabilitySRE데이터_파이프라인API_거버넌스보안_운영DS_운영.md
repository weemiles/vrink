## 1) Observability / SRE 운영 — SLO·알람·장애 대응

### 1.1 서비스 SLO(기본값)
- [운영 가드레일] 핵심 API(로그인/결제/핵심 조회)는 SLO를 최소 2개로 고정
    - Availability(성공률): ≥ 99.9% / 30일
    - Latency(p95): ≤ 300ms / 30일

### 1.2 에러 버짓(Release gate)
- [운영 가드레일] 30일 에러 버짓 소진율이 아래를 넘으면 “기능 릴리즈 동결”
    - ≥ 50%: 신규 기능 릴리즈는 승인 필요
    - ≥ 80%: 신규 기능 릴리즈 중단(버그/성능/안정성만)

### 1.3 알람 설계(알람 피로도 방지)
- [운영 가드레일] 알람은 아래를 만족하지 않으면 생성 금지
    - 15분 내 “액션(조치)”이 정의되어 있음
    - 1주 동안 false positive가 20%를 넘지 않음
- [운영 가드레일] 알람 쿨다운
    - 동일 알람은 15분 내 재알림 금지(중복 스팸 방지)

### 1.4 장애 대응 시간 기준(기본)
- [운영 가드레일] 심각도 분류
    - SEV0: 결제/로그인 전면 불가, 데이터 손상
    - SEV1: 핵심 플로우 부분 불가, 광범위 성능 저하
    - SEV2: 일부 기능/일부 세그먼트 영향
- [운영 가드레일] 목표 응답/복구
    - SEV0: ACK 5분, 완화(Mitigate) 30분
    - SEV1: ACK 10분, 완화 2시간
    - SEV2: ACK 30분, 완화 24시간

### 1.5 포스트모템 품질 기준
- [운영 가드레일] SEV0/1은 48시간 이내 포스트모템 완료
- [운영 가드레일] 재발 방지 액션아이템은 최소 2개 이상(모니터링 1개 + 예방 1개)

## 2) 데이터 파이프라인/분석 운영 — SSOT·DQ·지표 거버넌스

### 2.1 SSOT(Single Source of Truth) 규칙
- [운영 가드레일] “공식 지표”는 반드시 단일 모델/테이블에서만 계산
    - 대시보드가 직접 raw event를 집계하는 패턴 금지

### 2.2 데이터 품질(DQ) 게이트
- [운영 가드레일] ETL/모델 배포 시 아래 3개는 필수 검증
    - row count 변화율: 기준 대비 ±20% 초과 시 실패
    - null ratio(핵심 컬럼): > 1%면 실패
    - unique key 중복률: > 0.1%면 실패

### 2.3 이벤트 스키마 변경 감지
- [운영 가드레일] 이벤트 스키마 변경(필드 추가/삭제/타입 변경) 발생 시
    - 24시간 이내 영향 지표 목록 자동 생성
    - 영향 지표가 10개 이상이면 릴리즈 승인 필요

### 2.4 실험/수익 지표 정합성
- [운영 가드레일] 실험 주요 지표는 최소 1개 “비즈니스 지표”와 함께 고정
    - 예: 전환율 + 결제 성공률(또는 환불율)
- [운영 가드레일] 실험 종료 후 결과는 7일 내 기록(재현 가능한 쿼리/필터 포함)

## 3) API/스키마 거버넌스 — 버저닝·호환성·폐기

### 3.1 하위 호환(Backward compatibility) 규칙
- [운영 가드레일] 프로덕션 API에서 “breaking change”는 금지
    - 필드 삭제 금지
    - 필드 타입 변경 금지
    - enum 값 삭제 금지

### 3.2 Deprecation(폐기) 정책
- [운영 가드레일] 폐기 공지 기간
    - 내부/모바일 앱 연동: 90일
    - 외부 파트너/공개 API: 180일
- [운영 가드레일] 폐기 대상 엔드포인트는
    - 호출 비율이 전체의 0.5% 미만이 되는 시점부터 실제 제거 가능

### 3.3 계약 테스트(Contract test)
- [운영 가드레일] 핵심 API는 서버/클라이언트 계약 테스트를 필수로 둔다.
    - 실패하면 배포 차단

## 4) 보안 운영 — 취약점 SLA·비밀키·서드파티 SDK

### 4.1 취약점 대응 SLA(기본)
- [운영 가드레일] 심각도 기준(예시)
    - Critical: 원격 코드 실행/계정 탈취/PII 대량 노출
    - High: 인증 우회 가능성/권한 상승
    - Medium: 제한적 영향
- [운영 가드레일] 패치 SLA
    - Critical: 72시간
    - High: 7일
    - Medium: 30일

### 4.2 비밀키/인증서 운영
- [운영 가드레일] 모든 secret은 주기적 회전
    - API key: 90일
    - 인증서/서명키: 180일
- [운영 가드레일] secret 접근 권한은 기본 최소화
    - 접근 승인 시 만료 30일 필수

### 4.3 서드파티 SDK 공급망 리스크
- [운영 가드레일] 신규 SDK 도입 시 체크리스트
    - 네트워크 목적지(도메인) 목록 제출
    - 수집 데이터 항목 명시(PII 포함 여부)
    - 업데이트 주기/보안 공지 채널 확인
- [운영 가드레일] 보안/프라이버시 리스크로 분류되면
    - 릴리즈 전 “기능 플래그로 OFF 가능” 상태를 의무화

## 5) 조직 레벨 DS 운영 — 기여/릴리즈/마이그레이션

### 5.1 DS 변경 등급(Release tier)
- [운영 가드레일] 변경을 3등급으로 분류
    - Tier A: 토큰/공통 컴포넌트 변경(전 제품 영향)
    - Tier B: 일부 컴포넌트/패턴 변경
    - Tier C: 문서/예시/가이드 수정

### 5.2 릴리즈 정책
- [운영 가드레일] Tier A/B는 고정 릴리즈 주기
    - 월 1회
- [운영 가드레일] Tier A는 항상 마이그레이션 가이드를 포함
    - 마이그레이션 완료 목표: 90일

### 5.3 Breaking change 방지(DS)
- [운영 가드레일] DS는 “기본값 변경”을 breaking으로 취급
    - Tier A/B에서 기본값 변경 시 반드시 플래그/옵션으로 단계적 이행

### 5.4 기여(Contribution) 품질 기준
- [운영 가드레일] 신규 컴포넌트/변경 PR은 최소 4종 세트를 포함
    - 사용 가이드(언제/왜)
    - 상태(loading/disabled/empty/error 중 해당)
    - 접근성 체크(라벨/포커스/키보드)
    - 다크모드/토큰 준수 확인
