## 1) 관측(Observability) 실전 운영 규칙

### 1.1 프론트(앱/RN/WebView) 로그 레벨/볼륨
- [운영 가드레일] 프로덕션 로그 레벨
    - 기본: INFO 이하만 업로드
    - DEBUG/TRACE: 업로드 금지 (원격 플래그로도 15분 이상 지속 금지)
- [운영 가드레일] 1 세션당 로그 업로드 상한
    - 텍스트 로그: 200줄
    - 에러 스택: 최대 20건
    - 초과 시: 초과분은 로컬에서만 보관하고 업로드 차단

### 1.2 Error / Crash 알림 정책(노이즈 억제)
- [운영 가드레일] 알림은 “개별 에러”가 아니라 “집계 지표”로 발송한다.
    - 예: 에러율, 크래시 프리, p95 로딩, 주요 API 실패율
- [운영 가드레일] 에러 알림 트리거(권장 기본값)
    - 5분 윈도우에서 에러율 1% 초과
    - 또는 15분 윈도우에서 에러율 0.5% 초과
    - 또는 신규 릴리즈(24h)에서 크래시 프리 세션 99.5% 미만
    - 트리거 후 30분 쿨다운 (동일 원인 반복 알림 방지)

### 1.3 Trace/Span(분산 추적) 네이밍 + 필수 태그
- [운영 가드레일] trace는 “사용자 행동 단위”로 시작한다.
    - 예: checkout_submit, login_submit, search_submit
- [운영 가드레일] span 네이밍 규칙
    - UI: ui.<screen>.<action>
    - API: api.<service>.<endpoint>
    - Storage: storage.<type>.<op>
- [운영 가드레일] 모든 주요 span에 아래 태그를 강제
    - release_version, platform, screen_id, network_type, session_id
- [운영 가드레일] 샘플링
    - 기본: 1%
    - 신규 릴리즈 24시간: 5%
    - 인시던트 대응 중: 10% (최대 2시간, 이후 원복)

### 1.4 화면 성능 SLO(UX 기반)
- [운영 가드레일] “핵심 화면”은 3개 지표를 고정
    - p50 로딩, p95 로딩, 실패율
- [운영 가드레일] 기본 SLO(권장)
    - p50: 1.0s 이하
    - p95: 2.5s 이하
    - 실패율: 0.5% 이하
- [운영 가드레일] 위반 시 우선순위
    - p95 위반(느린 꼬리) > 실패율 위반 > p50 위반

## 2) 앱·웹뷰 업데이트/호환성 운영

### 2.1 최소 지원 버전(Min Supported Version) 정책
- [운영 가드레일] 최소 지원 버전 갱신 주기: 8주
- [운영 가드레일] 강제 업데이트(하드 블록) 조건
    - 보안 이슈, 결제/정산 오류, 개인정보 이슈
    - 또는 크래시/치명 결함으로 핵심 흐름 불가
- [운영 가드레일] 단계적 차단(소프트→하드)
    1) 1주: 배너 안내(건너뛰기 가능)
    2) 2주: 진입 시 모달(건너뛰기 가능)
    3) 3주: 핵심 기능 진입 시 차단(업데이트 필요)
    4) 4주: 앱 전체 차단(하드 블록)

### 2.2 WebView 캐시/리소스 버저닝 규칙
- [운영 가드레일] WebView 번들은 app_version + web_bundle_version 조합으로 버저닝
- [운영 가드레일] 캐시 무효화(강제 리프레시) 조건
    - 번들 버전 증가
    - feature flag 스키마 변경
    - 인증/권한 스키마 변경
- [운영 가드레일] 캐시 무효화 빈도 상한
    - 동일 사용자 기준: 24시간 1회 이하
    - 초과 시: 네트워크 비용/로딩 저하 이슈로 간주하고 원인 분석

### 2.3 Server-driven UI(스키마 호환 계약)
- [운영 가드레일] 클라이언트는 “모르는 필드”를 무시해야 한다.
- [운영 가드레일] 서버는 “필드 삭제” 대신
    - deprecated: true로 최소 2 릴리즈 유지 후 삭제
- [운영 가드레일] 필수 필드 누락 시 fallback
    - 화면 렌더링 실패 대신 Result/Empty로 강등(사용자 행동 경로 제공)

## 3) 결제/정산 이후 ‘사후 운영’(분쟁/환불/재처리)

### 3.1 상태 모델(최소 상태 집합)
- [운영 가드레일] 결제/주문 엔티티는 최소 아래 상태를 가진다.
    - authorized(승인/인증 완료)
    - captured(매입/결제 확정)
    - cancel_requested(취소 요청 접수)
    - canceled(취소 완료)
    - refund_requested
    - refunded
    - chargeback_in_progress
    - chargeback_won / chargeback_lost

### 3.2 Idempotency(중복 처리 방지) 규칙
- [운영 가드레일] “돈이 움직이는” 요청은 전부 idempotency key를 사용
    - 키 형식: <user_id>:<action>:<yyyy-mm-dd>:<random>
- [운영 가드레일] 키 TTL(재사용 방지 기간): 24시간
- [운영 가드레일] 동일 키로 재요청 시
    - 서버는 동일 결과를 반환
    - 신규 결제/환불 실행 금지

### 3.3 분쟁/CS 대응 SLA(운영 수치)
- [운영 가드레일] 1차 응답 SLA: 24시간 이내
- [운영 가드레일] 해결/중간결과 업데이트: 72시간 이내
- [운영 가드레일] 증빙(로그/이벤트) 보존 기간
    - 결제 관련 핵심 이벤트: 180일
    - 일반 행동 로그: 30~90일(정책에 맞춰 선택)

### 3.4 환불/재처리 안전장치
- [운영 가드레일] 부분환불은 1건당 최대 3회까지만 허용(운영 실수 방지)
- [운영 가드레일] 재처리(retry) 정책
    - 결제/환불 API 실패 시 자동 재시도: 최대 2회
    - 재시도 간격: 1분 → 10분
    - 3번째 실패부터는 수동 처리 큐로 이동

## 4) 보안 심화(Threat modeling + 사고 대응)

### 4.1 Threat model ‘필수 체크리스트’(릴리즈 게이트)
- [운영 가드레일] 아래 6개 질문에 “예”가 하나라도 있으면 보안 리뷰 필수
    1) 권한/역할이 바뀌는가?
    2) 결제/환불/정산 로직이 바뀌는가?
    3) PII(이메일/전화/식별자)를 저장/전송하는가?
    4) 관리자 기능/운영 도구가 추가되는가?
    5) 외부 연동(Webhook/3rd party)이 추가되는가?
    6) 파일 업로드/다운로드가 추가되는가?

### 4.2 비밀/키 관리(운영 규칙)
- [운영 가드레일] 클라이언트 앱에 영구 비밀키(Private key) 포함 금지
- [운영 가드레일] 토큰/키 회전 주기
    - 액세스 토큰: 15분
    - 리프레시 토큰: 30일
    - 서버 API 키(운영): 90일

### 4.3 보안 사고 대응 런북(타임라인 기준)
- [운영 가드레일] 탐지 후 0~30분
    - 영향 범위 확인(사용자 수, 기능, 지역)
    - 임시 완화(기능 플래그 off, 레이트리밋 강화)
- [운영 가드레일] 30~120분
    - 재현/원인 분류
    - 로그/증거 보존(변조 방지)
    - 임시 패치 또는 롤백 결정
- [운영 가드레일] 24시간 이내
    - 고객 공지 초안/FAQ 준비(필요 시)
    - 재발 방지 작업 항목화(테스트/게이트 추가)

## 5) Backoffice(관리자 도구) 표준 운영

### 5.1 역할/권한 모델(최소 3단계)
- [운영 가드레일] 최소 권한 레벨을 3단계로 고정
    - viewer(조회만)
    - operator(일반 처리)
    - admin(권한/정책 변경)
- [운영 가드레일] 고위험 액션은 4-eyes(2인 승인)
    - 환불(부분/전체)
    - 정산 수동 조정
    - 사용자 계정 잠금/해제
    - 정책/한도 변경

### 5.2 관리자 액션 감사로그(Audit Log) 최소 스키마
- [운영 가드레일] 모든 관리자 액션은 아래 필드를 필수로 저장
    - actor_id, actor_role, action_name, target_type, target_id
    - before, after(가능하면 diff)
    - ip, user_agent, timestamp_ms, request_id
- [운영 가드레일] 감사로그 보존 기간: 1년(365일)

### 5.3 운영자 실수 방지 UX(강제 규칙)
- [운영 가드레일] destructive 액션은 2단계 확인
    - 1차: 경고 모달
    - 2차: 대상 식별자 일부 입력(예: 마지막 4자리)
- [운영 가드레일] “되돌리기(Undo)” 제공 원칙
    - 가능하면 5분 이내 undo 제공
    - 불가하면 “원복 절차(플레이북 링크)”를 UI에 노출

### 5.4 운영 화면 품질 기준
- [운영 가드레일] 모든 리스트 화면은 아래를 기본 제공
    - 검색(키워드)
    - 필터(상태/기간)
    - 정렬(최신/우선순위)
    - CSV 내보내기(PII 마스킹 적용)
