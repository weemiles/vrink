## 1) 보안/리스크 심화 — 위협 모델링 + 사고가 나는 지점 체크

### 1.1 위협 모델링(Threat Model) 최소 템플릿
- 기능 1개당 최소 6항목을 체크합니다.
    1) 보호해야 하는 자산(데이터/돈/계정)
    2) 공격자(일반 사용자/탈취 계정/내부자/봇)
    3) 공격 벡터(피싱/리플레이/스크래핑/세션 하이재킹)
    4) 피해(금전/개인정보/신뢰)
    5) 탐지(로그/알림)
    6) 완화(제한/차단/추가 인증)

### 1.2 인증/로그인 하드 룰(수치)
- 로그인 실패 제한:
    - 동일 계정 기준 5회 연속 실패 → 10분 잠금
    - 동일 IP 기준 20회/10분 초과 → 추가 캡차/차단(택1)
- 비밀번호/인증 변경 후 세션 처리:
    - 변경 성공 시 기존 세션은 즉시 폐기(전부)
- 고위험 행동(비밀번호 변경/출금/계좌 변경) 재인증:
    - 마지막 재인증 이후 5분이 지나면 재인증 요구

### 1.3 리플레이/중복 요청 방지(수치)
- 결제/송금/주문 생성 같은 “중복되면 돈이 새는” 요청은 다음을 고정합니다.
    - idempotency key 필수
    - 동일 key 재요청은 24시간 동안 동일 결과 반환
    - 서버에서 key별 처리 상태를 기록하고, 실패/성공 모두 남김

### 1.4 민감정보 노출 방지(규칙)
- 화면/로그/분석에 실명·전화·주민번호·계좌번호 원문 저장 금지
- 표시 원칙(최소 마스킹):
    - 전화번호: 가운데 4자리 마스킹
    - 계좌번호: 뒤 4자리만 노출
> 표기 수치/형식은 제품 운영 심화(수치) — 내비게이션·비동기/캐시·표기(Formatting)·보안 UX·Analytics의 마스킹 규칙을 따르되, 여기서는 “원문 저장 금지” 운영 원칙을 추가합니다.

### 1.5 ‘사회공학(CS 피싱)’ 방지 규칙
- CS가 계정 정보를 바꿀 수 있는 경우, 다음을 고정:
    - 전화/채팅만으로 이메일/비번/계좌 변경 금지
    - 본인 확인은 최소 2요소(예: 앱 내 인증 + 보조 확인)로만
    - 예외 처리(긴급)도 로그 + 승인 2인(CS 1 + 리드 1)

## 2) 개인정보/규제/컴플라이언스 운영 — 데이터 수명주기 규칙

### 2.1 데이터 분류(최소 4등급)
- D0: 공개 가능(마케팅 카피 등)
- D1: 내부용(운영 지표, 익명화)
- D2: 개인정보(PII)
- D3: 고위험 개인정보(정부ID, 계좌/카드, 건강 정보 등)

### 2.2 수집 최소화 규칙(수치)
- 신규 기능이 PII를 수집해야 하면:
    - “필수” PII는 최대 3필드
    - 4필드 이상은 ‘필수/선택’ 분리 + 수집 목적을 필드 옆에 표시

### 2.3 보관기간/파기 기본값(운영 가드)
- 기본 보관기간을 먼저 고정하고, 필요 시에만 늘립니다.
    - 인증 로그: 90일
    - 서비스 사용 로그(행동 이벤트): 180일
    - 고객 문의/CS 티켓: 1년
- 파기 원칙:
    - 기간 만료 시 “소프트 삭제” 금지, 영구 삭제
    - 백업 포함 파기 범위 정의(최소 1회/분기 점검)

### 2.4 로그/이벤트의 PII 혼입 방지 규칙
- 이벤트/로그 payload에 다음이 들어가면 빌드/배포 차단(원칙)
    - 이메일(@ 포함)
    - 전화번호(010 패턴)
    - 계좌번호/카드번호(긴 숫자열)
- 샘플링:
    - 고빈도 이벤트는 기본 10% 샘플링
    - 장애/결제/인증 이벤트는 100%

### 2.5 배포 전 리뷰 패키지(체크리스트)
- 개인정보/보안 리뷰는 “패키지 8항목”으로 제출합니다.
    1) 수집 항목 목록
    2) 목적/법적 근거
    3) 저장 위치(서버/클라/로그)
    4) 보관기간
    5) 파기 방식
    6) 제3자 제공 여부
    7) 접근 권한(누가 볼 수 있는지)
    8) 사용자 고지 문구

## 3) 관측/장애 대응(Observability + Incident) — SLO/알림/플레이북

### 3.1 SLO 최소 세트(기능별)
- 기능 1개당 SLO는 최소 2개를 고정합니다.
    1) 가용성(성공률)
    2) 지연시간(p95)

### 3.2 기본 임계값(수치, 출발점)
- 성공률(availability): 99.5%/일 미만이면 경보
- 지연시간(latency): p95 2.0초 초과이면 경보
- 결제/인증은 더 엄격:
    - 성공률 99.8%/일
    - p95 1.5초

### 3.3 알림 피로도(Noise) 제한
- 한 기능에서 알림은 30분에 최대 1회
- 동일 원인 알림은 2시간 내 반복 금지(집계로 묶음)

### 3.4 인시던트 등급(Severity) 규칙
- SEV1: 결제/로그인 불가, 또는 사용자 5% 이상 영향
- SEV2: 핵심 기능 일부 실패, 또는 사용자 1% 이상 영향
- SEV3: 우회 가능, 제한적 영향

### 3.5 대응 타임라인 목표(수치)
- SEV1:
    - 탐지(TTD) 5분 이내
    - 완화(TTM) 30분 이내 (롤백/차단/우회)
- SEV2:
    - TTD 15분
    - TTM 2시간

### 3.6 사후 회고(Postmortem) 최소 템플릿
- 사고 1건당 7항목을 반드시 기록:
    1) 영향 범위(사용자 수/시간)
    2) 루트 원인
    3) 탐지 실패/성공 포인트
    4) 임시 조치
    5) 영구 조치
    6) 재발 방지 체크(알림/SLO/테스트)
    7) 오너 + 기한

## 4) 고객지원(CS) 관점 제품 설계 — 재현 가능성과 문의 감소

### 4.1 CS 티켓 분류(최소 6종)
- 결제
- 인증/로그인
- 계정/개인정보
- 배송/진행 상태
- 버그/오류
- 기타

### 4.2 재현을 위한 ‘지원 코드’ 규칙(수치)
- 모든 주요 플로우(결제/인증/신청)는 지원 코드를 발급합니다.
    - 길이: 8~12자
    - 형식: 대문자+숫자 혼합
    - 유효기간: 30일
- 지원 코드로 CS가 확인 가능한 정보는 최소화:
    - 상태(성공/실패)
    - 실패 사유 코드
    - 타임스탬프
    - 개인 식별 정보는 기본 비노출

### 4.3 문의 감소 UI 규칙
- 에러 화면에는 다음을 기본 포함(최대 2개까지만)
    - ‘다시 시도’(1)
    - ‘문의하기’(2)
- 문의하기를 노출하는 기준(수치):
    - 동일 사용자가 같은 에러를 3회 연속 겪으면 문의 CTA 노출

### 4.4 환불/취소/분쟁 플로우 운영 규칙
- 취소 가능 여부는 상태로 명확히 분기:
    - 가능/불가능/조건부(수수료)
- 취소/환불은 결과 통지 시간을 고정:
    - 처리 시작 즉시 알림
    - 완료 예상 시간: 최대 3영업일 같은 상한 문구를 기본 제공

## 5) 온보딩/리텐션(성장) — 퍼널/메시징/코호트 규칙

### 5.1 Activation 정의(규칙)
- Activation은 ‘가입’이 아니라 핵심 가치 행동 1회 완료로 정의
- 핵심 행동은 1개만(최대 1)
    - 예: 첫 결제 완료, 첫 예약 완료, 첫 기록 저장

### 5.2 온보딩 단계 수 제한(수치)
- 첫 가치 도달까지 화면 수:
    - 권장 3~5 화면
    - 6 화면 이상이면: 1) 건너뛰기, 2) 나중에 하기, 3) 점진적 프로필 수집 중 최소 1개를 포함

### 5.3 메시지(푸시/인앱) 피로도 제한(수치)
- 푸시:
    - 사용자 1인당 주 3회 기본 상한
    - 결제/보안 알림은 예외(필수)
- 인앱 배너/팝업:
    - 동일 캠페인은 세션당 1회
    - 사용자가 닫으면 7일 재노출 금지

### 5.4 리텐션 코호트 해석 최소 규칙
- 최소 3코호트를 매주 확인합니다.
    - 신규(첫 주)
    - 4주차 유지
    - 휴면 복귀
- 개선 판단 기준(대략 수치):
    - D1/D7 리텐션이 +2%p 이상이면 유의미 후보
    - 단, 시즌성/캠페인이 있으면 같은 주의 전년/전월과 함께 비교

### 5.5 성장 실험의 ‘가드레일’
- 성장 실험도 Guardrail은 반드시 둡니다.
    - unsubscribe(푸시 해지) +0.3%p 이상이면 중단
    - 리뷰/문의 급증(동일 키워드 10건/일) 시 중단

## 6) 적용 순서(최소 실행 경로)
1. 기능 1개를 골라 Threat Model 6항목을 채운다.
1. 데이터 분류(D0~D3)와 보관기간 기본값을 정한다.
1. 결제/인증에 SLO 2개(성공률, p95)를 붙인다.
1. 지원 코드(8~12자, 30일)를 결제/인증 플로우에 추가한다.
1. Activation 1개를 정의하고, 첫 가치까지 3~5 화면을 넘지 않게 줄인다.
