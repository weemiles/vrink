## 1) 결제/정산 도메인 심화 — 상태 모델/재시도/대사 규칙

### 1.1 거래(결제) 상태 표준(최소 7상태)
- [운영 가드레일] 거래 단위(결제/취소/환불)는 최소 아래 상태로 내부 모델을 고정
    - created (요청 생성)
    - authorized (승인)
    - captured (매입)
    - failed (실패)
    - canceled (승인취소)
    - partially_refunded (부분환불)
    - refunded (전액환불)
- [운영 가드레일] UI에는 상태를 그대로 노출하지 말고, 사용자 메시지는 3그룹으로만 매핑
    - 성공
    - 진행 중(지연 포함)
    - 실패(재시도 가능/불가로만 분기)

### 1.2 “중복 결제 방지” 운영 기준
- [운영 가드레일] 동일 사용자 + 동일 상품/금액 + 동일 결제수단의 결제 요청은
    - 30초 윈도우 내에는 “동일 요청”으로 간주해 idempotency key 강제
- [운영 가드레일] 결제 버튼은 탭 1회 후 즉시 비활성화, 복구는 아래 중 1개
    - 성공/실패 응답 수신
    - 또는 10초 타임아웃 후 “상태 확인” CTA로 전환

### 1.3 재시도/타임아웃 정책
- [운영 가드레일] 결제 API 타임아웃은 클라이언트 기준
    - 연결/응답 대기 합산 10초를 상한
- [운영 가드레일] 자동 재시도는 최대 1회만 허용
    - 그 이후는 사용자 액션(재시도 버튼)으로만

### 1.4 대사(Reconciliation) 운영 규칙
- [운영 가드레일] 결제/환불/취소는 D+1(다음날)까지 “최종 확정 지연” 가능성을 전제
    - 사용자 화면에는 “확정 전” 배지를 최소 24시간 유지할 수 있어야 함
- [운영 가드레일] 대사 불일치 탐지는 매일 1회(권장: 새벽 배치)
    - 불일치율이 0.1%를 넘으면 운영 알림(슬랙/메일 등) 발송

## 2) 푸시/알림 운영 — 빈도/품질/실험 가드레일

### 2.1 발송 빈도 상한(피로도 방지)
- [운영 가드레일] 마케팅/프로모션 알림
    - 사용자당 일 최대 1건
    - 사용자당 주 최대 3건
- [운영 가드레일] 트랜잭션 알림(결제, 보안 등)
    - 상한 없음. 단, 동일 이벤트 중복 발송 방지 윈도우 60초

### 2.2 조용한 시간(Do-not-disturb) 규칙
- [운영 가드레일] 로컬 타임 기준 21:00~08:00에는
    - 마케팅 알림 발송 금지
    - 단, 보안/사기/결제 실패 등 “중요도 P0”는 예외

### 2.3 품질 게이트(릴리즈 차단 기준)
- [운영 가드레일] 푸시 발송 기능 변경 시 아래 실패하면 릴리즈 차단
    - 옵트아웃(수신 거부) 사용자에게 발송되는 케이스 1건이라도 발생
    - 딥링크 실패율(앱 열었는데 목적지 이동 실패)이 1% 초과

### 2.4 실험(A/B) 운영 규칙
- [운영 가드레일] 알림 실험은 최소 2지표를 고정
    - 1차: open rate
    - 2차: 목표 행동 전환율
- [운영 가드레일] 실험 기간은 최소 7일 또는 표본 1,000명(둘 중 큰 값)

## 3) 검색/추천 — ML 운영(버전/드리프트/롤백)

### 3.1 모델/랭킹 버전 운영 규칙
- [운영 가드레일] 모델/랭킹은 배포 시 항상
    - model_version (예: semver)
    - feature_version
    - index_version
    를 로그에 남겨 “재현 가능” 상태를 만든다.

### 3.2 롤백 기준(Release rollback)
- [운영 가드레일] 배포 후 2시간 이내 아래 조건 중 1개면 자동 롤백
    - 검색 결과 0건 비율이 기준 대비 +0.5%p 이상 증가
    - CTR이 기준 대비 -5% 이상 하락
    - 오류율(5xx/timeout)이 기준 대비 +1%p 이상 증가

### 3.3 드리프트(Drift) 감지 규칙
- [운영 가드레일] 입력 분포가 변하는 피처는 주 1회 이상 드리프트 체크
    - top-N 카테고리 분포에서 Jensen-Shannon distance 0.1 초과 시 경보

### 3.4 오프라인 평가셋 운영
- [운영 가드레일] 오프라인 평가셋은 최소 월 1회 업데이트
    - 샘플 500쿼리 이상
    - “헤드/토르소/롱테일” 쿼리 비율을 50/30/20으로 유지

## 4) 데이터 거버넌스/프라이버시 — 스키마/보존/마스킹

### 4.1 PII 차단(수집 단계) 규칙
- [운영 가드레일] 이벤트/로그 스키마에 PII 필드는 원칙적으로 금지
    - 예외: 해시/토큰화된 식별자만 허용
- [운영 가드레일] 클라이언트 로그에는 아래를 절대 남기지 않는다.
    - 전체 이메일
    - 전체 전화번호
    - 주민번호/계좌번호 원문

### 4.2 보존/파기 정책(기본값)
- [운영 가드레일] 원본 이벤트 로그 보존 기간 기본값
    - 90일
- [운영 가드레일] 보안/부정거래 조사 목적 로그
    - 1년
- [운영 가드레일] 삭제 요청(계정 삭제) 처리 SLA
    - 요청 수신 후 30일 이내 파기 완료

### 4.3 접근 제어(Least privilege)
- [운영 가드레일] 운영자/분석가 권한은 최소 단위로 분리
    - PII 접근 권한은 기본 OFF
    - 예외 승인 시 만료(30일)를 반드시 설정

## 5) 백오피스(운영자 도구) UX 표준 — 위험 작업/권한/로그

### 5.1 위험 작업 등급(P0~P2) 정의
- [운영 가드레일] 운영자 액션을 3등급으로 분류해 UI/승인/로그를 강제
    - P0: 금전/계정 잠금/권한 변경/대량 삭제
    - P1: 환불/정책 변경/배너 노출
    - P2: 단순 조회/메모/태그

### 5.2 2인 승인(4-eyes) 기준
- [운영 가드레일] P0는 항상 2인 승인
    - 요청자 ≠ 승인자
    - 승인 TTL은 30분. 만료되면 재승인 필요

### 5.3 대량 작업 안전장치
- [운영 가드레일] 100건 초과 영향을 주는 작업은
    - 미리보기(영향받는 대상 수) 필수
    - 실행 전 “확인 텍스트 입력” 필수(예: DELETE 100+)
    - 실행 후 10분 내 취소(undo) 가능한 구조를 우선 설계

### 5.4 감사 로그(Audit log) 최소 필드
- [운영 가드레일] 모든 P0/P1 액션은 아래 필드를 반드시 기록
    - actor_id
    - action_type
    - target_ids(최대 1,000개까지)
    - before/after diff(가능한 범위)
    - ip / user_agent
    - timestamp

## 6) 최소 도입 순서(바로 적용)
1) 결제 idempotency 30초 윈도우(1.2)
2) 알림 빈도 상한(2.1) + 야간 금지(2.2)
3) 검색/추천 롤백 기준(3.2)
4) 로그 PII 금지 + 90일 보존(4.1~4.2)
5) 백오피스 P0 2인 승인 + 감사로그 필드 고정(5.2~5.4)
