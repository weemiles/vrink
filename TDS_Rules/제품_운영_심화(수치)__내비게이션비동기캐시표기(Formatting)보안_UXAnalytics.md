# 제품 운영 심화(수치) — 내비게이션·비동기/캐시·표기(Formatting)·보안 UX·Analytics


## 0. 이 페이지의 목적 (중복 방지)
- 이 페이지는 TDS 컴포넌트의 Props/기본값/스펙을 다시 쓰지 않습니다.
- 또한 이미 만들어둔 “TDS 실무 운영 규칙(수치 중심)” 문서(폼/선택/피드백/A11y/토큰)는 재작성하지 않고, 그 다음 레벨의 운영 규칙만 다룹니다.
- 아래 기존 페이지들과 겹치지 않도록 주제를 분리합니다.
    - 컴포넌트 스펙/수치(버튼/타이포/컬러/레이아웃): TDS 헌법 — Toss 모바일 앱 디자인 실무 가이드라인 (2026): 레이아웃·타이포·컬러·컴포넌트·접근성
    - 오버레이/토스트/에셋/로딩/Skeleton/TableRow: TDS 심화 학습 — 오버레이·토스트·에셋·스켈레톤·TableRow (원문 구조 기반)
    - SearchField/Loader/Menu/ListRow/Keypad/Carousel 등: TDS 컴포넌트 실무 노트 2 — SearchField·Loader·Menu/Dropdown·ListRow·Keypad·Carousel·NumericSpinner·BottomInfo
    - TextButton/IconButton/Result/Slider: TDS 컴포넌트 실무 노트 3(Mobile) — TextButton·IconButton·Result·Slider
    - ProgressBar/Stepper/ListFooter/ProgressStepper/GridList: TDS 컴포넌트 실무 노트 4(Mobile) — ProgressBar·Stepper·ListFooter·ProgressStepper·GridList
    - 폼/선택/피드백/A11y/토큰 운영 수치: TDS 화면 운영 규칙(수치) — Form·Selection·Feedback·A11y·토큰 적용

## 1. 내비게이션/화면 구조(IA) — 수치로 고정하는 규칙

### 1.1 화면 제목/행동의 밀도(상단 영역)
- 상단 영역(타이틀 + 액션)은 한 화면에서 다음을 넘기지 않습니다.
    - 우측 아이콘 액션: 최대 2개
    - 상단 주요 행동(텍스트 버튼/링크): 최대 1개
- 이유: 3개를 넘으면 행동 우선순위가 흐려지고, 접근성 라벨 관리도 급격히 어려워짐.

### 1.2 뒤로가기(Back) 규칙
- 뒤로가기로 화면을 벗어날 때 “데이터 손실 가능”이 있으면, 확인 UI를 띄우되 기준을 수치화합니다.
    - 변경점(Dirty) 판단: 입력값/선택값이 초기값 대비 1개라도 다르면 dirty로 간주
    - 경고 표시 기준:
        - 저장되지 않은 변경이 있고,
        - 변경 후 경과 시간이 2초 이상인 경우(실수 터치 완충)
- 저장 완료 후 자동 뒤로가기 delay: 300ms
    - 너무 즉시 이동하면 저장이 “안 된 느낌”을 주는 경우가 많음.

### 1.3 리스트 → 상세 → 수정 플로우 최소 규칙
- 리스트에서 상세로 진입할 때:
    - skeleton/loader는 TDS 심화 학습 — 오버레이·토스트·에셋·스켈레톤·TableRow (원문 구조 기반) 규칙을 따르되, “상세 화면 상단(타이틀)”은 최소 1줄을 먼저 고정해서 레이아웃 점프를 줄입니다.
- 수정 화면에서 완료 후 돌아가기:
    - 성공 토스트를 쓸 경우: 토스트 노출 후 뒤로가기까지 600ms
    - Result 페이지를 쓸 경우: 자동 이동 금지, 사용자가 버튼으로 종료(기본)

### 1.4 딥링크 진입(중간 화면부터 시작)
- 딥링크로 중간 화면 진입 시 “이전 화면 스택이 비어있음” 처리 원칙:
    - back 버튼이 의미가 없으면, back을 숨기기보다 닫기(close) 아이콘으로 대체
    - close를 누르면 복귀 목적지를 2단계로 고정:
        1) 앱 홈(탭 루트)
        2) 불가능하면(권한/로그인) 로그인 화면

## 2. 비동기/에러/재시도/캐시 — 운영 수치

### 2.1 네트워크 타임아웃(UX 기준)
- “요청 시작 → 아무 반응도 없음” 기준으로 다음을 고정합니다.
    - 0.2초(200ms) 이하: 로딩 미표시 권장(깜빡임 방지)
    - 2초: “느림”으로 간주, 문구/스켈레톤으로 안정감 제공
    - 7초: 사용자에게 명시적 안내(‘처리 중’ + 취소/뒤로) 제공
    - 12초: 실패 처리 + 재시도 버튼 제공(기본)
> 0.2초/7초/12초는 TDS 화면 운영 규칙(수치) — Form·Selection·Feedback·A11y·토큰 적용에 이미 있는 값이지만, 여기서는 비동기 운영 전체에 적용 범위를 확장합니다(컴포넌트/폼 영역 재작성은 아님).

### 2.2 재시도(Manual retry) 규칙
- 버튼 재시도는 무한 허용이 아니라 다음으로 제한합니다.
    - 동일 요청(같은 파라미터) 연속 실패 시: 최대 3회까지
    - 3회 실패 후: “연결 상태 확인” 안내 + 고객센터/문의 링크(있다면)
- 재시도 버튼 연타 방지:
    - 클릭 후 1.0초 동안 disabled

### 2.3 자동 재시도(Automatic retry) 규칙
- 자동 재시도는 사용자에게 비용이 적은 요청만 허용(예: 단순 조회)
- 횟수/간격(고정 제안)
    - 1차: 0.5초
    - 2차: 1.5초
    - 3차: 3.0초
    - 총 3회 후 실패 처리

### 2.4 캐시 전략(UX)
- “캐시된 화면 + 백그라운드 갱신”을 기본으로 두는 경우:
    - 캐시 유효시간(stale) 기본값: 60초
    - 60초가 지나면 화면은 보여주되, 상단에 조용한 갱신 표시(로더/스피너)를 최대 2초만 노출
    - 2초를 넘으면: 갱신 표시를 제거하고 결과만 반영(무한 스피너 금지)

### 2.5 오프라인(Offline) 처리
- 오프라인 감지 시 즉시 알림을 띄우지 말고, “실패한 순간”에만 안내
- 오프라인 안내는 같은 세션에서 최대 1회만 토스트/배너로 노출(스팸 방지)

## 3. 데이터 표기(Formatting System) — 숫자/날짜/단위 규칙

### 3.1 금액(KRW) 표기
- 기본: 천 단위 구분(,) 적용
- 단위 표기:
    - 원화 단독 표시: 1,234원
    - 큰 금액 압축(선택):
        - 10,000원 이상부터 만원 단위 검토
        - 표시 규칙: 소수점 1자리까지만(반올림)
            - 예: 12.3만원
            - 예: 12만원(소수점 0이면 생략)
- 부호/색상 규칙은 토큰 기반(여기서 색상 재정의하지 않음)

### 3.2 퍼센트(%)
- 기본 소수점 자리수:
    - 일반 지표: 0자리 (예: 12%)
    - 정밀 지표(금리/수수료): 1자리 (예: 12.3%)
    - 1% 미만: 최대 2자리 (예: 0.25%)
- 반올림은 일반적으로 round(사사오입)

### 3.3 날짜/시간
- 기본 포맷:
    - 날짜: YYYY.MM.DD
    - 날짜+시간: YYYY.MM.DD HH:mm
- 상대시간(‘방금’류) 사용 기준:
    - 1분 미만: 방금
    - 1분~59분: N분 전
    - 1시간~23시간: N시간 전
    - 24시간 이상: 절대 날짜로 전환

### 3.4 전화번호/계좌번호 마스킹(표기)
- 전화번호(010-xxxx-xxxx): 가운데 4자리 마스킹
    - 예: 010-\*\*\*\*-1234
- 계좌번호: 뒤 4자리만 노출
    - 예: \*\*\*\*-\*\*\*\*-1234

## 4. 보안/프라이버시 UX — 수치/제한

### 4.1 인증 코드(OTP/SMS) 재전송 정책
- 재전송 가능 시점: 최초 요청 후 30초
- 재전송 최대 횟수: 5회/시간
- 잘못된 코드 입력 제한:
    - 연속 실패 5회 → 10분 잠금(또는 서버 정책에 맞춤)

### 4.2 세션 만료/자동 로그아웃
- 앱 비활성(백그라운드) 상태에서 자동 잠금 기준: 5분
    - 금융/고위험 화면이면 1분도 검토
- 재인증 후 복귀:
    - 원래 화면으로 복귀는 허용하되, 민감정보(잔액/개인정보)는 재렌더링 시 다시 가림

### 4.3 캡처/공유 UX 가이드
- 민감정보가 포함된 화면에서 “공유”가 필요하면
    - 기본 공유 이미지에서 민감정보는 기본 마스킹
    - 사용자가 ‘표시’ 버튼을 눌러야만 원문 노출(기본값은 가림)

## 5. Analytics / 실험(A|B) — 이벤트 설계 수치

### 5.1 이벤트 네이밍/파라미터 최소 세트
- 이벤트는 “행동 단위”로만 남깁니다(렌더/노출 남발 금지).
- 기본 이벤트 5종만 고정:
    1) screen_view
    2) click
    3) submit
    4) success
    5) fail
- 각 이벤트에 공통으로 넣는 파라미터 최소 세트:
    - screen_name
    - component_name
    - action
    - result

### 5.2 중복 로깅 방지(수치)
- 같은 버튼을 연속 클릭할 때:
    - 첫 클릭 이후 0.5초 내 발생한 동일 click 이벤트는 버립니다(디바운스)
- 스크롤 기반 노출 이벤트(필요 시):
    - 동일 아이템 노출은 세션당 1회만 기록(기본)

### 5.3 A|B 실험 기간/샘플(운영 기준)
- 최소 실험 기간: 7일
- 표본이 적은 기능이면: 14일
- 실험 중 UI 변경(추가 개편)은 금지(결과 해석 불가능해짐)

[ ] 금액/날짜/퍼센트 표기 규칙이 화면마다 다르지 않다.
[ ] 인증 재전송/실패 제한 수치가 문구와 실제 동작이 일치한다.
[ ] 이벤트가 중복으로 과다 발생하지 않는다(디바운스/세션 1회).
