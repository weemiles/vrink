# 제품 품질/운영 규칙(수치) — QA·변경관리·카피·i18n·성능 UX


## 0. 범위와 중복 방지 선언
- 이 페이지는 컴포넌트 스펙/Props/기본값을 다루지 않습니다. (컴포넌트별 내용은 기존 페이지 참고)
- 또한 아래 문서에 이미 정리된 “운영 수치”와 같은 항목을 재작성하지 않습니다.
    - TDS 기본/컴포넌트 수치: TDS 헌법 — Toss 모바일 앱 디자인 실무 가이드라인 (2026): 레이아웃·타이포·컬러·컴포넌트·접근성
    - 오버레이/토스트/에셋/스켈레톤/표: TDS 심화 학습 — 오버레이·토스트·에셋·스켈레톤·TableRow (원문 구조 기반)
    - SearchField/Loader/Menu/ListRow 등: TDS 컴포넌트 실무 노트 2 — SearchField·Loader·Menu/Dropdown·ListRow·Keypad·Carousel·NumericSpinner·BottomInfo
    - TextButton/IconButton/Result/Slider: TDS 컴포넌트 실무 노트 3(Mobile) — TextButton·IconButton·Result·Slider
    - Progress/Stepper 계열: TDS 컴포넌트 실무 노트 4(Mobile) — ProgressBar·Stepper·ListFooter·ProgressStepper·GridList
    - 폼/선택/피드백/A11y/토큰 운영 수치: TDS 화면 운영 규칙(수치) — Form·Selection·Feedback·A11y·토큰 적용
    - 내비게이션/비동기/표기/보안/Analytics 운영 수치: 제품 운영 심화(수치) — 내비게이션·비동기/캐시·표기(Formatting)·보안 UX·Analytics
- 여기서는 위 문서들에서 다루지 않은, 제품 품질을 운영으로 고정시키는 규칙(프로세스 + 수치)만 적습니다.

## 1) 디자인 QA (릴리즈 전/후) — 케이스를 “조건 조합”으로 고정

### 1.1 QA 커버리지 규칙(최소 세트)
- 각 화면은 최소 12케이스를 통과해야 배포 가능(기본값).
- 12케이스는 아래 3축 조합으로 만듭니다.
    - 네트워크: 정상 | 느림(2~7초) | 실패(즉시/타임아웃) = 3
    - 사용자 상태: 로그인 | 비로그인/세션만료 = 2
    - 접근성: 기본 텍스트 | 큰 텍스트(최대) = 2
    - 총 3×2×2 = 12
> 느림/타임아웃 수치 자체는 제품 운영 심화(수치) — 내비게이션·비동기/캐시·표기(Formatting)·보안 UX·Analytics에 이미 있으므로, 이 페이지에서는 QA 케이스 생성 규칙만 고정합니다.

### 1.2 “큰 텍스트(최대)” 합격 기준(수치)
- 큰 텍스트(최대)에서 다음 4개가 한 화면에서 0건이어야 합격입니다.
    1) 터치 가능한 요소가 화면 밖으로 밀림
    2) 필수 CTA가 접히거나 스크롤 끝에서만 보임
    3) 본문이 2줄 이상 겹침(줄간격 붕괴)
    4) 버튼/탭/칩의 라벨이 의미가 바뀔 정도로 잘림
- 잘림은 허용하되 기준을 둡니다.
    - 라벨이 12자 이상이면: 최대 2줄까지 허용
    - 2줄에서도 잘리면: 줄임표 처리 + 툴팁/보조 텍스트로 의미 보완(택1)

### 1.3 모션/애니메이션 QA (Reduce Motion)
- Reduce Motion 환경에서 합격 기준:
    - “필수 정보(성공/실패/경고)” 전달이 애니메이션에 의존하지 않는다.
    - 화면 전환/오버레이 열림 닫힘 애니메이션이 제거되어도 레이아웃 점프가 1회 이하다.

### 1.4 스냅샷 기준(디자인 회귀 탐지)
- 변경 PR 1개당 최소 스냅샷:
    - iOS 1종 + Android 1종 = 2장/화면
    - 라이트/다크가 있으면 ×2 → 4장/화면
- 스냅샷은 “대표 케이스”만 찍습니다.
    - 정상 네트워크
    - 로그인 상태
    - 기본 텍스트

### 1.5 릴리즈 후 핫픽스 기준(수치)
- 다음 중 1개라도 해당되면 당일 핫픽스를 기본 원칙으로 합니다.
    - 결제/인증/회원가입 플로우에서 이탈 증가(전일 대비) +30%
    - 크래시율 0.5% 초과
    - 서버 5xx로 인해 실패율 2% 초과(동일 기능)

## 2) 컴포넌트/토큰 변경 관리 — “회귀”를 숫자로 막는 운영

### 2.1 변경 분류(Severity) 규칙
- 변경은 무조건 아래 3단계로 라벨링합니다.
    - S1: 시각만 변경(의미/레이아웃 불변)
    - S2: 레이아웃/간격/줄바꿈 영향
    - S3: 인터랙션/접근성/의미 변화(또는 API 변경)

### 2.2 배포 게이트(필수 조건)
- S1
    - 스냅샷 1세트(라이트)만으로 통과 가능
- S2
    - 스냅샷 + QA 12케이스 중 최소 6케이스 통과 필수
- S3
    - QA 12케이스 전부 통과
    - 다크모드 포함(있다면)
    - 접근성(스크린리더 라벨) 점검 1회 필수

### 2.3 영향 범위 산정 규칙
- 변경된 토큰/컴포넌트가 영향 주는 화면 수를 계산해서 우선순위를 정합니다.
    - 영향 화면 수 ≥ 10: 즉시 롤아웃 금지, 단계적 배포(5%→25%→100%)
    - 영향 화면 수 3~9: QA 확대(12→18케이스)
    - 영향 화면 수 ≤2: 기본 QA(12케이스)

### 2.4 단계적 배포(롤아웃) 기본 스텝
- 기본 스텝: 5% → 25% → 100%
- 각 스텝 관찰 시간(최소): 2시간
- 중단 기준:
    - 실패율이 직전 스텝 대비 +1%p 증가
    - 문의/리뷰가 단시간에 5건 이상 증가(동일 키워드)

## 3) 콘텐츠/카피 시스템 — 문장을 시스템으로 만들기

### 3.1 문장 길이 제한(수치)
- 제목(Title): 한글 기준 최대 18자
- 본문(Description): 한글 기준 최대 60자, 기본은 2줄 내
- 버튼 라벨:
    - 주요 CTA: 2~6자
    - 보조 CTA: 2~8자
- 토스트: 최대 20자

### 3.2 금지 패턴(규칙)
- 모호한 표현 금지(대체 표현을 고정)
    - “잠시 후” → “최대 N초” 또는 “다시 시도”
    - “문제가 발생” → 원인 카테고리(네트워크/권한/입력/서버)
- 사용자 책임으로 읽히는 표현 금지
    - “잘못하셨습니다” 류 대신 “다시 확인해주세요”

### 3.3 에러 문구 템플릿(고정)
- 구조는 항상 3파트(순서 고정)
    1) 무엇이 안 됐는지(결과)
    2) 왜(원인 카테고리)
    3) 다음 행동(재시도/설정/문의)
- 예시 템플릿:
    - 제목: 요청을 완료할 수 없어요
    - 설명: 네트워크 연결이 불안정해요. 다시 시도해주세요.
    - 버튼: 다시 시도

### 3.4 성공 문구 규칙
- 성공은 감탄/과장을 쓰지 않고, 상태 변화를 말합니다.
    - “완료!” 대신 “저장했어요”, “신청했어요”, “변경했어요”
- 성공 후 다음 행동이 있으면:
    - 버튼 1개만 제공(최대 1)
    - 버튼 라벨은 명사형 금지, 동사형으로(예: “확인하기”)

### 3.5 로그/분석을 위한 카피 ID 규칙
- 문구는 화면마다 ID를 가집니다.
    - 포맷: screenName.sectionName.messageName
    - 예: transfer.error.network_timeout
- 카피 수정은 텍스트만 바꾸지 말고 ID 버전을 올립니다.
    - v1 → v2

## 4) i18n / l10n — 텍스트 길이와 포맷을 “깨지지 않게” 설계

### 4.1 확장률(Expansion) 가드
- 한국어 기준으로 레이아웃을 잡더라도 다음을 통과해야 합니다.
    - 영어 길이 확장: +35%
    - 독일어 길이 확장: +50%
- 버튼/탭 라벨은 확장률을 더 보수적으로 잡습니다.
    - 버튼 라벨: +25%
    - 탭 라벨: +20%

### 4.2 줄바꿈 규칙(고정)
- 타이틀: 2줄까지 허용, 2줄에서 끝나지 않으면 줄임표
- 버튼: 2줄 금지(기본). 반드시 1줄
    - 1줄 불가하면: 라벨을 줄이거나 UI를 바꾼다(버튼 폭 늘리기 금지부터 검토)

### 4.3 숫자/날짜 포맷 로케일 규칙
- 포맷팅 로직은 “UI 컴포넌트”가 아니라 “포맷터”로 분리
- 천 단위 구분은 로케일에 따라 자동(예: 1,234 vs 1.234)
- 시간 표기(12/24h)는 로케일 설정을 따르고, 화면 내 혼용 금지

### 4.4 RTL(오른쪽→왼쪽) 대비 규칙
- 아이콘(화살표/이동)은 RTL에서 방향 반전 필요
- 정렬은 leading/trailing 기반으로 잡고 left/right 하드코딩 금지

## 5) 퍼포먼스 UX — “체감 속도”를 규칙으로 고정

### 5.1 프레임/응답성 목표(수치)
- 스크롤/드래그: 60fps 목표(최소 50fps)
- 탭 전환/필터 적용:
    - 200ms 이내면 로딩 표기 생략 가능
    - 200ms~700ms는 최소 로더(점/스피너) 1개
    - 700ms 이상이면 스켈레톤/플레이스홀더로 레이아웃 고정
> 200ms 같은 임계값은 이미 일부 문서에서 언급되지만, 여기서는 “성능 UX 결정 규칙”으로만 사용합니다.

### 5.2 이미지 로딩 규칙
- 리스트/피드의 썸네일:
    - 첫 화면(Above the fold) 기준 최대 12개만 즉시 로드
    - 나머지는 스크롤 진입 시 로드
- 썸네일 기본 전략:
    - 저해상도(blur) → 고해상도 업그레이드(2단계)

### 5.3 프리패치(Preload) 기준
- 다음 화면 진입 확률이 높으면 프리패치 허용
    - CTR(클릭률) 30% 이상인 카드/리스트 아이템
- 프리패치는 동시에 최대 2개 요청까지만

### 5.4 캐시/갱신 UX 규칙(중복 방지)
- 캐시 유효시간/백그라운드 갱신 수치 자체는 제품 운영 심화(수치) — 내비게이션·비동기/캐시·표기(Formatting)·보안 UX·Analytics에 이미 있으므로, 여기서는 “갱신이 사용자에게 보이는 방식”만 고정합니다.
    - 갱신 중 표시(스피너/로더)가 화면을 흔들지 않는다.
    - 갱신 실패 시 “조용한 실패”를 기본으로 하고, 사용자 행동이 필요한 경우에만 배너/토스트를 띄운다.

## 6) 이 페이지 적용 순서(최소 실행 경로)
1. 릴리즈 전에 “QA 12케이스”를 각 주요 화면에 붙인다.
1. 변경 작업은 S1/S2/S3로 라벨링한다.
1. 버튼/타이틀/토스트 길이 제한을 린트처럼 운영한다.
1. 영어 +35%, 독일어 +50% 확장 테스트를 대표 화면 3개에 먼저 돌린다.
1. 피드/리스트 화면에서 ‘첫 화면 이미지 12개 즉시 로드’ 제한을 적용한다.
