## 1) 모션/애니메이션 운영 규칙 (Reduce Motion 포함)

### 1.1 모션 사용 기준
- “상태 변화”를 설명하는 목적일 때만 사용합니다.
    - 로딩 → 완료
    - 확장/접기
    - 리스트 삽입/삭제(되돌리기 포함)
- “주의 분산”이 리스크인 화면(결제, 인증, 금전/개인정보 입력)에서는 장식 모션 금지.

### 1.2 기본 모션 시간(수치)
- 탭/버튼 pressed 피드백: 80~120ms
- 작은 전환(토글, 펼침/접기): 150~200ms
- 화면 전환/오버레이 전환: 200~300ms
- 로딩→완료(성공/실패) 강조: 300~450ms

### 1.3 이징(easing) 규칙
- 기본: ease-out(빠르게 시작, 부드럽게 멈춤)
- 반복/무한 로딩: linear 또는 일정 속도로만(가속/감속 반복 금지)

### 1.4 모션 거리/크기 제한(수치)
- 마이크로 인터랙션 이동 거리: 최대 12px
- 오버레이 등장 이동: 최대 24px
- 스케일 애니메이션: 0.98~1.02 범위

### 1.5 Reduce Motion(접근성) 대응 규칙
- prefers-reduced-motion: reduce 또는 OS 모션 감소가 켜져 있으면:
    - 모든 이동/확대 축소 애니메이션 OFF
    - opacity 전환만 허용(권장 150ms)
    - 무한 애니메이션은 정적 상태로 대체

### 1.6 모션 QA 체크리스트
[ ] 중요한 CTA/금액/에러 메시지가 모션 중 가려지지 않는다.
[ ] 모션 감소 설정에서 기능이 동일하게 동작한다.
[ ] 로딩/완료 모션이 ‘성공처럼 보이는데 실패’ 같은 오해를 만들지 않는다.

## 2) 딥링크/푸시/알림 설계 — 실패/권한/버전 차이를 운영 규칙으로 고정

### 2.1 딥링크 기본 정책
- 모든 딥링크는 “최종 목적지”만 표현하고, 중간 화면을 강제하지 않습니다.
- 딥링크가 실패하면 다음 우선순위로 폴백합니다.
    1) 앱 내 홈(또는 탭 루트)
    2) 해당 기능의 랜딩 페이지
    3) 웹 대체 페이지

### 2.2 딥링크 실패 원인별 UX 규칙
- 로그인 필요:
    - 로그인 후 원래 목적지로 자동 복귀(1회)
- 권한 필요:
    - 권한을 주지 않아도 가능한 ‘보기 모드’가 있으면 그 모드로 진입
- 앱 버전 미지원:
    - 업데이트 유도 화면으로 이동
    - 업데이트가 불가한 환경이면 웹 대체

### 2.3 푸시 빈도 상한(수치)
- 마케팅/리텐션 푸시: 사용자 1인당 주 3회
- 제품 공지/운영 공지: 주 1회
- 보안/결제/인증 알림: 예외(필수)

### 2.4 푸시 발송 시간대 규칙
- 기본 발송 시간대: 09:00~20:00
- 야간(20:00~09:00)은 ‘긴급/보안’만 허용

### 2.5 푸시 메시지 길이(가드레일)
- 제목: 최대 20자
- 본문: 최대 50자
- CTA는 1개만(‘확인’, ‘지금 보기’ 등)

### 2.6 알림 권한 요청 타이밍(운영 규칙)
- 설치 직후 요청 금지
- “가치 행동” 1회 완료 이후 요청(예: 저장/결제/예약 완료 후)
- 권한을 거절한 뒤 재요청은 7일 후부터

## 3) 오프라인/네트워크 불안정 UX — 실패를 ‘상태 머신’으로 다루기

### 3.1 상태 정의(최소)
- Online
- Offline
- SyncPending(로컬 저장 후 동기화 대기)
- Syncing
- SyncFailed

### 3.2 재시도(backoff) 기본값(수치)
- 즉시 재시도: 1회
- 이후 재시도 간격: 1s → 2s → 4s → 8s → 16s
- 최대 간격 상한: 30s
- 자동 재시도 최대 횟수: 5회
- 5회 실패 후:
    - “수동 재시도” 버튼 제공
    - 실패 원인 1줄(네트워크/서버/권한)

### 3.3 오프라인에서도 가능한 행동(규칙)
- 작성/편집이 핵심인 기능은:
    - 오프라인에서도 로컬 저장을 허용
    - SyncPending 상태를 노출
- 금전/인증/민감 변경은:
    - 오프라인에서 실행 금지
    - 대신 ‘대기열’에 쌓지 말고, 온라인 복귀 후 다시 시도하도록 안내

### 3.4 에러 메시지 규칙
- 항상 다음 요소를 포함(최소 2개)
    - 현재 상태(오프라인/지연)
    - 사용자가 할 수 있는 다음 행동(재시도/나가기)

## 4) 권한/온디바이스 기능 UX — 거절/영구거절을 시나리오로 고정

### 4.1 권한 프리-프롬프트(사전 설명) 규칙
- 다음 조건이면 프리-프롬프트를 먼저 보여줍니다.
    - 권한이 없으면 기능을 사용할 수 없음
    - 사용자가 ‘왜 필요한지’ 예상하기 어려움(위치/연락처)
- 프리-프롬프트는 3문장 이내:
    1) 무엇을 위해
    2) 어떤 데이터가
    3) 언제 사용되는지

### 4.2 권한 거절 상태별 대응
- 1회 거절:
    - 즉시 재요청 금지
    - 화면 내 대체 경로 제시
- 영구 거절(설정에서만 허용):
    - 설정 이동 CTA 제공
    - 설정 이동은 세션당 최대 1회만 노출(피로도 방지)

### 4.3 권한이 없어도 되는 ‘대체 경로’ 규칙
- 위치 권한이 없으면:
    - 수동 입력(주소/지역) 제공
- 카메라 권한이 없으면:
    - 갤러리 업로드(가능하면) 제공

## 5) A11y 확장: 스크린리더 ‘실사용’ 시나리오 테스트 세트
> 기존 A11y 규칙에 “실제 테스트 케이스”를 추가로 고정합니다.

### 5.1 폼(입력) 시나리오
[ ] 필수 입력 누락 시, 스크린리더가 에러를 읽는다.
[ ] 에러 위치로 포커스가 이동한다.
[ ] 에러 해소 후 상태 변화가 과도하게 반복 낭독되지 않는다.

### 5.2 선택 컨트롤 시나리오
[ ] SegmentedControl/Radio 그룹명이 먼저 읽힌다.
[ ] 현재 선택 상태(checked)가 올바르게 읽힌다.
[ ] disabled 옵션이 구분되어 읽힌다.

### 5.3 내비게이션/구조 시나리오
[ ] 헤딩 구조가 최소 2단(페이지 제목/섹션) 존재한다.
[ ] 주요 CTA는 탭 이동으로 10회 이내에 도달 가능하다.

### 5.4 차트/시각화 시나리오
[ ] 차트는 ‘요약 문장 1개’ + ‘세부 데이터 보기’ 경로를 제공한다.
[ ] 세부 데이터는 표/리스트로 제공한다.

## 6) Server-driven UI / Feature Flag 운영 규칙

### 6.1 플래그 우선순위 규칙
- 기본값(Default) → 실험(Experiment) → 긴급 차단(Kill switch)
- Kill switch는 항상 최우선

### 6.2 롤아웃 단계(수치)
- 0.5% (내부) → 5% → 25% → 50% → 100%
- 각 단계는 최소 24시간 관찰 후 다음 단계로

### 6.3 실패 시 롤백 기준(가드레일)
- crash-free users가 -0.2%p 이상 악화되면 롤백
- 핵심 전환(Activation/결제 등)이 -1%p 이상 악화되면 롤백
- CS/리뷰 키워드가 하루 10건 이상 급증하면 롤백

### 6.4 설정 기반 UI의 일관성 규칙
- 같은 상태(예: 가입 완료)에서
    - 플래그에 따라 버튼 위치/명칭이 바뀌지 않게 한다
- 바뀌어야 한다면:
    - 실험군/대조군 모두 동일한 이벤트 스키마를 사용

## 7) 적용 순서(최소 실행 경로)
1. 모션 시간(150/200/300ms)과 Reduce Motion 대응을 코드/디자인에 고정한다.
1. 딥링크 폴백(홈→랜딩→웹)과 알림 빈도(주 3회)를 운영 규칙으로 고정한다.
1. 오프라인 재시도(1→2→4→8→16, max 30s, 5회)와 실패 후 수동 재시도 UX를 만든다.
1. 권한 프리-프롬프트(3문장)와 거절/영구거절 정책(7일/세션 1회)을 적용한다.
1. 스크린리더 테스트 세트를 QA 체크리스트에 붙인다.
1. 플래그 롤아웃(0.5→5→25→50→100, 24h)과 롤백 기준을 운영한다.
