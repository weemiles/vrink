## 1) 데이터/스키마 마이그레이션 운영(모바일 관점)

### 1.1 스키마 버전 정책
- [운영 가드레일] 스키마 버전은 정수 1씩 증가시키고, 릴리즈 노트에 반드시 기록한다.
- [운영 가드레일] 클라이언트는 최소 최근 2개 버전의 마이그레이션을 “연속 실행”할 수 있어야 한다.
    - 예: v12 → v14 직접 업그레이드 가능

### 1.2 마이그레이션 실행 시간/실패 예산
- [운영 가드레일] 앱 부팅 시 마이그레이션은 p95 300ms 이내를 목표로 한다.
    - 초과 시: 백그라운드 마이그레이션(지연 실행)로 전환
- [운영 가드레일] 마이그레이션 실패율(릴리즈 24h): 0.1% 이하
    - 초과 시: 즉시 롤백/핫픽스 우선순위 1

### 1.3 로컬 DB 마이그레이션 실패 시 복구 플로우
- [운영 가드레일] 실패 시 자동 재시도: 최대 1회
- [운영 가드레일] 재시도 후에도 실패하면, 아래 3단계 중 1개를 선택해 “정책으로 고정”
    1) 읽기 전용 모드(핵심 기능 제한) + 고객 안내
    2) 로컬 데이터 초기화(로그인 재요구) + 고객 안내
    3) 서버 동기화 기반 복원(가능한 경우)
- [운영 가드레일] 어떤 선택을 하든, 사용자에게는:
    - “무엇이 불가능한지”
    - “다음 행동(재로그인/업데이트/문의)”
    를 2줄 내로 안내한다

### 1.4 점진적 마이그레이션(dual read/write) 규칙
- [운영 가드레일] 위험한 스키마 변경은 dual write 기간을 둔다.
    - 기간: 최소 1 릴리즈(2주~4주)
- [운영 가드레일] dual write 종료 조건
    - 신규 스키마 필드 유실률 0.01% 이하
    - 서버 검증(샘플) 오류 0건/7일

## 2) 실험(Experimentation) 통계·해석 운영

### 2.1 실험 사전 등록(Pre-registration) 최소 필드
- [운영 가드레일] 실험 시작 전에 아래 6개는 확정하고 기록한다.
    - Primary metric 1개
    - Guardrail metric 2개(예: 크래시/환불/CS)
    - 실험 단위(사용자/세션/노출)
    - 최소 실행 기간(일)
    - 대상 트래픽 비율
    - 롤백 조건(수치)

### 2.2 최소 실행 기간/트래픽 비율
- [운영 가드레일] 최소 실행 기간: 7일 (요일 효과 제거)
- [운영 가드레일] 최소 트래픽: 10%
    - 결제/보안/정산 등 고위험 실험은 1% → 5% → 10% 3단계 램프업

### 2.3 SRM(표본비율불일치) 게이트
- [운영 가드레일] SRM 체크는 매일 1회 자동 실행
- [운영 가드레일] SRM 의심 기준
    - 기대 비율 대비 편차가 ±2%p 초과 AND 24시간 지속
    - 또는 특정 OS/버전에서만 편차가 ±3%p 초과
- [운영 가드레일] SRM 의심 시: 결과 해석 중단 + 배정 로직/이벤트 누락부터 점검

### 2.4 조기 종료/중도 롤백 규칙
- [운영 가드레일] 조기 성공 선언 금지: 최소 기간(7일) 전에는 성공으로 종료하지 않는다.
- [운영 가드레일] 즉시 롤백 조건(하나라도 만족)
    - Guardrail metric 악화: +10% 이상 (상대 변화)
    - 크래시 프리 세션: 99.5% 미만
    - 결제/환불 실패율: +20% 이상

### 2.5 다중 비교(동시 실험) 운영
- [운영 가드레일] 동일 Primary metric을 건드리는 동시 실험은 최대 2개까지만 허용
- [운영 가드레일] 동일 사용자에게 동시에 배정되는 실험 수 상한: 3개

## 3) 푸시/인앱메시지/CRM 운영(전송·빈도·품질)

### 3.1 채널 우선순위
- [운영 가드레일] 동일 메시지의 채널 우선순위(기본)
    1) 인앱
    2) 푸시
    3) 이메일
    4) 문자

### 3.2 빈도 제한(Frequency capping)
- [운영 가드레일] 사용자 기준 기본 상한
    - 푸시: 하루 2건, 주 7건
    - 인앱: 하루 3회 노출, 주 12회 노출
    - 이메일: 주 3건
- [운영 가드레일] “트랜잭션 메시지(결제/보안)”는 상한에서 제외하되, 동일 이벤트 중복 발송은 금지

### 3.3 조용한 시간(Quiet hours)
- [운영 가드레일] 기본 Quiet hours: 21:00~08:00 (로컬 타임존)
- [운영 가드레일] Quiet hours에 보낼 수 있는 예외
    - 보안 알림
    - 결제/정산 알림
    - 장애/중요 공지

### 3.4 중복/지연/실패 방지 키
- [운영 가드레일] 모든 발송은 de-dup key를 가진다.
    - 키: <user_id>:<campaign_id>:<event_id>
    - TTL: 7일
- [운영 가드레일] 지연 발송 유효 기간(expiry)
    - 프로모션: 24시간
    - 보안/결제: 즉시 (지연 큐에 남아있으면 폐기)

### 3.5 옵트아웃/권한 상태 정책
- [운영 가드레일] 푸시 권한 거부 상태에서 “권한 요청”을 다시 띄우는 최소 간격: 30일
- [운영 가드레일] 옵트아웃 사용자는:
    - 마케팅 메시지 발송 금지
    - 트랜잭션 메시지만 허용

## 4) 국가/결제수단/플랫폼 확장 운영(국내/해외)

### 4.1 로케일/통화/시간 규칙
- [운영 가드레일] 모든 금액 표시는 “정수 최소 단위(원/센트)”로 저장하고, 표시 단계에서만 포맷한다.
- [운영 가드레일] 시간은 저장 시 UTC 고정, 표시 시 로컬 타임존 변환.
- [운영 가드레일] 로케일 확장 시 필수 QA 항목
    - 통화 기호 위치
    - 천 단위 구분자
    - 소수점 자리수(0/2)
    - 날짜 포맷(YYYY-MM-DD vs MM/DD)
    - 주소 입력(필드 수/순서)

### 4.2 결제수단별 실패/리트라이 정책 차등
- [운영 가드레일] 결제 실패 재시도는 “수단별”로 상한을 둔다.
    - 카드/간편결제: 즉시 1회만 자동 재시도
    - 계좌이체/가상계좌: 자동 재시도 금지(사용자 재시도 유도)
- [운영 가드레일] 재시도 시도 횟수는 UI에 노출하지 않는다(보안/악용 방지)

### 4.3 규제/정책 변경 대응 프로세스(시간 기준)
- [운영 가드레일] 정책 변경(수수료/세금/약관) 감지 후
    - 7일 이내: 영향 분석 + 변경 범위 확정
    - 14일 이내: 스테이징 반영 + QA
    - 21일 이내: 프로덕션 반영
- [운영 가드레일] 긴급(법/보안) 변경은 72시간 내 반영을 목표로 한다.

## 5) 모바일 네트워크 품질/통신사/환경 이슈 대응

### 5.1 네트워크 타임아웃/재시도 기본값(클라이언트)
- [운영 가드레일] 요청 타임아웃
    - 일반 API: 10초
    - 업로드/다운로드: 30초
- [운영 가드레일] 자동 재시도(서버 부하/중복 방지 고려)
    - GET/멱등 요청: 최대 2회
    - POST/돈이 움직이는 요청: 자동 재시도 금지(서버 idempotency 전제 시에만 1회)
    - 백오프: 0.5s → 2s

### 5.2 네트워크 전환/오프라인 전이 UX
- [운영 가드레일] 오프라인 감지 후
    - 1초 이내: 배너(“네트워크가 불안정해요”)
    - 10초 지속: 핵심 CTA 비활성 + 재시도 버튼 제공
- [운영 가드레일] 재시도 버튼은 1분 내 최대 3회까지만 허용(폭주 방지)

### 5.3 캡티브 포털/프록시 환경
- [운영 가드레일] 로그인/결제 같은 핵심 흐름에서
    - 연속 실패 2회 발생 시: “네트워크 환경(공용 Wi‑Fi 등) 안내”를 노출

### 5.4 환경 이슈 수집(재현 가능성 강화)
- [운영 가드레일] 버그 리포트에 아래 8개는 자동 첨부(사용자 동의 전제)
    - app_version
    - device_model
    - os_version
    - locale
    - network_type
    - carrier(if available)
    - timestamp
    - last_screen_id
