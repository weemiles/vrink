## 1) Theming / 다크모드 / 고대비 운영 규칙

### 1.1 테마 전환의 기본 원칙
- [TDS 명시] 토큰 기반으로 색을 관리하고, 하드코딩 색을 피한다. (토큰/시맨틱 구조는 기존 문서 참조)
- [운영 가드레일] 테마 전환은 “화면 단위”가 아니라 앱 전역에서 일관되게 동작해야 하며, 특정 화면만 테마가 다르면 결함으로 처리한다.

### 1.2 대비(contrast) 최소 기준(수치)
- [운영 가드레일]
    - 본문 텍스트: 최소 4.5:1
    - 18px 이상(또는 굵은 큰 글씨): 최소 3:1
    - 아이콘(정보 전달 목적): 최소 3:1

### 1.3 다크모드에서 이미지/일러스트 처리 규칙
- [운영 가드레일]
    - 로고/일러스트가 “검정 텍스트 포함”이면 다크모드 전용 에셋을 별도로 준비한다.
    - 다크모드에서 배경이 어두워져 “윤곽이 사라지는” PNG는:
        - 외곽선 1px(중립색) 또는
        - 약한 그림자(blur 8px 이내) 또는
        - 배경 패치(rounded 8~12px)
        중 하나로 가시성을 보장한다.

### 1.4 차트/상태색(semantic) 충돌 방지 규칙
- [운영 가드레일]
    - 같은 화면에서 semantic 색(성공/에러 등)은 최대 2개 상태색까지만 동시에 노출.
    - 3개 이상 상태를 보여야 하면:
        - 색 1개 + 모양(아이콘) + 라벨 텍스트로 구분

### 1.5 고대비 모드 대응 규칙
- [운영 가드레일]
    - 고대비 모드에서:
        - 구분선은 최소 1px 유지
        - disabled는 투명도만으로 표현 금지(텍스트/아이콘 명도 차 + 라벨/설명으로 보완)

## 2) Form(입력) 운영 규칙 — 검증/마스킹/에러 표준화

### 2.1 검증 타이밍 정책(수치)
- [운영 가드레일]
    - 입력 중(타이핑 중): 형식 경고는 최소화
    - 포커스 아웃(blur): 형식 검증 1회
    - 제출 시(submit): 전체 검증 + 에러 요약

### 2.2 에러 메시지 길이/구조 규칙
- [운영 가드레일]
    - 에러 메시지는 1문장
    - 해결 행동은 1개 (예: “다시 입력해주세요”)
    - 같은 필드에서 에러는 1개만 보여준다(우선순위 적용)

### 2.3 에러 우선순위(규칙)
1. 필수 누락(Required)
1. 형식 오류(Format)
1. 범위 오류(Range)
1. 서버 거절(Server validation)

### 2.4 입력 길이 제한 기본값(권장 수치)
- [운영 가드레일]
    - 이름: 최대 20자
    - 검색어: 최대 50자 (기존 SearchField 예시와 일치)
    - 메모/요청사항: 최대 200자

### 2.5 마스킹/포맷 정책(수치)
- [운영 가드레일]
    - 전화번호: 3-4-4 또는 3-3-4로 자동 하이픈
    - 카드번호: 4-4-4-4 그룹, 표시 시 마지막 4자리만 노출
    - 생년월일: YYYY.MM.DD, 유효 범위(예: 1900~현재 연도)

### 2.6 제출 중 UX 규칙
- [TDS 명시] loading + disabled 동시 적용 패턴은 기존 문서 규칙을 따른다.
- [운영 가드레일] 제출이 2초를 넘으면:
    - 진행 상태 문구 1줄을 추가한다.
    - 취소가 가능한 작업이면 “취소”를 제공한다.

## 3) Chart/데이터 시각화 — 정의/접근성/해석 규칙

### 3.1 지표 정의 템플릿(필수 필드)
- [운영 가드레일] 모든 지표는 아래 5개를 문서로 고정합니다.
    1) 분자/분모
    2) 집계 윈도우(예: D1, D7)
    3) 포함/제외(취소/환불 포함 여부)
    4) 타임존 기준
    5) 갱신 주기

### 3.2 이상치/결측치 표기 규칙
- [운영 가드레일]
    - 결측(데이터 없음): “—”로 표시하고 툴팁에 사유(수집 중/권한)
    - 이상치(outlier): 점 색을 바꾸지 말고(의미 혼동), 마커(●) + 툴팁으로만 표시

### 3.3 경보(알림) 임계치 기본값(권장 수치)
- [운영 가드레일]
    - 핵심 지표가 전일 대비 -20% 이하 하락 → 경보
    - 에러율이 +1%p 이상 상승 → 경보
    - 결제 실패율이 +0.5%p 이상 상승 → 경보

### 3.4 스크린리더 요약 규칙
- [운영 가드레일] 차트는 다음 2개를 항상 제공.
    - 요약 문장 1개(예: “지난 7일 평균은 3.2%이고, 어제는 2.1%입니다.”)
    - “데이터 표로 보기” 경로

## 4) 결제/금융/리스크 UX — 사기 방지·실수 방지 운영 규칙

### 4.1 금액 입력 실수 방지(수치)
- [운영 가드레일]
    - 금액 입력은 기본적으로 3자리 콤마(또는 locale)로 즉시 포맷
    - 0이 연속으로 3개 이상 붙는 입력은(예: 1000000) “단위 확인”을 1회 띄운다.
    - 최대 금액 상한이 있으면 UI에 상한을 노출(예: “최대 500만원”).

### 4.2 최종 확인 화면(Review) 구성 규칙
- [운영 가드레일] 최종 확인에서는 반드시 다음 3개를 다시 보여준다.
    - 받는 대상(계좌/상점)
    - 금액
    - 수수료/총액(있다면)

### 4.3 취소/정정 가능 시간(권장 수치)
- [운영 가드레일]
    - 사용자가 실수했을 때의 “되돌리기” UX는 가능하면 10초 내에 제공(Undo 토스트/바텀시트 등)
    - 10초가 불가하면:
        - 취소 정책/지원 경로를 즉시 노출

### 4.4 사기/피싱 리스크 문구 규칙
- [운영 가드레일] 경고 문구는 2줄 이내, 행동은 1개(예: “취소하기”) + 보조 링크 1개(“자세히”)

## 5) Observability / 운영 관측 — 릴리즈·실험·장애 감지 규칙

### 5.1 핵심 운영 지표(최소 세트)
- [운영 가드레일] 아래 6개는 모든 릴리즈에서 기본 수집
    - crash-free users
    - API error rate
    - p95 latency
    - 핵심 전환율(Activation 또는 결제)
    - 결제 실패율(해당 서비스라면)
    - 고객 문의/리뷰 키워드(일 단위)

### 5.2 알림 임계치(권장 수치)
- [운영 가드레일]
    - p95 latency가 기준 대비 +30% 상승 → 경보
    - API error rate가 +1%p 상승 → 경보
    - crash-free users가 -0.2%p 하락 → 경보 (기존 롤백 기준과 일치)

### 5.3 릴리즈 관측 윈도우(수치)
- [운영 가드레일] 배포 후:
    - 30분: 장애 징후(에러율/크래시) 집중 모니터링
    - 24시간: 지표 안정성 확인 후 확장

### 5.4 이벤트 스키마 버전 관리 규칙
- [운영 가드레일]
    - 이벤트는 event_name + schema_version을 반드시 포함
    - breaking change는 버전 +1 후, 구버전 이벤트는 30일 유지

## 6) 최소 실행 경로(적용 순서)
1. 테마/다크모드: 대비 기준(4.5:1)과 이미지 처리 규칙을 QA 체크리스트에 넣는다.
1. Form: 검증 타이밍(blur 1회, submit 전체)과 에러 메시지 1문장 규칙을 고정한다.
1. Chart: 지표 정의 템플릿(5필드)과 접근성 요약+표 제공을 고정한다.
1. 금융/리스크: 최종 확인 3요소(대상/금액/수수료)와 Undo 10초 원칙을 고정한다.
1. Observability: 임계치(+30% latency, +1%p error, -0.2%p crash-free)와 관측 윈도우(30분/24h)를 운영한다.
